# Различения (ключевые термины)

> Загружается всегда. 1-2 строки на различение. Критерий: DP.KR.001 §6.
> Развёрнутые (тесты, примеры): `memory/hard-distinctions.md`. Авторские warm: `memory/distinctions-warm.md`.

- **IWE = Intellectual Work Environment** (не «Intelligent», не «Working»). Аналогия: IDE.
- **Развитие ≠ Образование.** Не использовать: «школа», «вуз», «студент», «преподаватель».
- **Скилл ≠ SKILL.md ≠ Роль ≠ Протокол.** Способность агента (мастерство) / спецификация-контракт / функциональное место / ритуал ОРЗ. Тест: скилл = умение играть, SKILL.md = партитура.
- **settings.json ≠ settings.local.json.** Проектный (hooks, git) vs персональный (permissions, .gitignore).
- **Профиль ошибок агента ≠ DayPlan.** Профиль повторяющихся ошибок — инструмент **для агента** (инжекция в системный промпт через UserPromptSubmit hook). DayPlan — артефакт **для пилота**. Тест: «кто читатель — пилот или агент?» (WP-356, 30 мая).
- **Static promote-checks ≠ Dynamic verify.** `validate-fmt-scripts.sh` ловит плейсхолдеры/личные пути/syntax. `verify-before-promote.sh` — runtime в env -i, smoke-кейсы, JSON-validation hook'ов. Перед commit в FMT-шаблон обязательны оба (WP-356, [[lessons_promote_pipeline_dynamic_verify]]).
- **Pack-знание ≠ Реализационное решение** (HD #29). Доменная истина → Pack. Технический выбор → DS.
- **Процесс ≠ Артефакт-результат ≠ Артефакт-процесс** (peer-session 2026-06-08, Цэрэн). Процесс = деятельность, не имеет материального выхода. РП может быть "Описание процесса миграции" (документ/артефакт-результат) или "Инженерия процесса установки" (методология/артефакт-процесс), но сам процесс как таковой = не РП. Тест: «это готовый результат (документ, система, методология) или чистая деятельность без выхода?» Грань: операционный конвейер (повторяющийся процесс) ≠ закрываемая фаза РП; конвейер держат и модернизируют, не закрывают как РП. Связано с [[Удержание нормы ≠ Поднятие нормы]] и [[Inventory ≠ Progress]].
- **OwnerIntegrity:** Один факт — одно место. Дублирование Pack↔DS = ошибка синхронизации.
- **WP-context ≠ Результат работы.** Context в DS-strategy, результат в целевом DS/Pack.
- **Персона ≠ Память ≠ Контекст** (HD #27, DP.D.052 v2). Критерий: writer + owner. Персона = **distributed composition** (Ory anchor + Git declarations + Neon refs), Память = Neon (Observed + Derived), Контекст/Проекция = runtime ephemeral. v2 (2026-05-31): таблица расщеплена на 4 оси — Writer / Identity-anchor / State-storage / Snapshot-unit.
- **Носитель ≠ Персона ≠ Декларация Персоны** (DP.D.052 §10, DP.ARCH.005 §0, 2026-05-31). Носитель = человек в физическом мире (вне платформы); Персона = entity-в-IWE как composition (anchor+decls+refs); Декларация = один Git snapshot. Тест: «удалю Git → пропадёт декларация, Персона остаётся (anchor живёт)». Адресует интуицию «в БД не Персона, а её Версия» без переименования. Lifecycle anchor: Pre-Grant `claim_token` → Granted `Ory subject_id`. Импорт Letta: наш Persona = их `human block`, **не** `persona block` (= self-описание агента).
- **Проблема ≠ Задача ≠ ФР ≠ Работа** (HD #24, DP.D.053). Ступор / метод известен / спецификация / физ.мир.
- **Система ≠ Роль** (D.141 PD). Носитель существует независимо; роль = функциональное место, переносима.
- **Система-в-роли ≠ Роль** (FPF A.2.1, R5, HD #26). Работу выполняет система-в-роли; роль лишь задаёт метод и контекстное участие. Нотация: `система#роль:контекст`. Роль сама ничего не делает.
- **Целевая система проекта ≠ класс систем** (FPF A.2.1, R5, HD #26). Целевая система проекта — роль, которую некоторая система играет в конкретном проекте. Одна физическая система в разных проектах = разные роли. Тест: «это класс вещей или функциональное место в контексте?»
- **Сервис ≠ метод** (FPF A.6.8, R5, HD #31). Сервис = обещание + провайдерская роль + метод оказания + работа оказания + свидетельства результата. Метод — один компонент сервиса. «Метод оказания» ≠ сервис целиком.
- **Мастерство (роль) ≠ Степень мастерства (характеристика)** (FPF A.2, R9, HD #32). Мастерство::роль = функциональная роль части организма/экзокортекса в деятельности. Степень мастерства = измеримая характеристика качества исполнения. Тест: «это кто исполняет или насколько хорошо?»
- **Экзоскелет ≠ Автопилот** (DP.D.046). Усиление мышления / LLM автономно. Тест: можешь объяснить без «ИИ подсказал»?
- **Лог ≠ Инцидент ≠ State file** (DP.D.049). Хронология у исполнителя / сбой+разбор в governance / снимок у исполнителя.
- **Скрипт ≠ Агент** (DP.D.048). Фиксированный flow / LLM выбирает шаг. Тест: уберу LLM — сломается?
- **Характеристика выпускника ≠ рычаг** (PD.FORM.093 §2). Выходное свойство (clarity, agency…) / механизм воздействия (M1/M2/M4/W из RCS, FORM.089).
- **Состояние объекта ≠ Стадия процесса** (граница PACK-personal / PACK-<домен>, 2026-05-22). Состояние = кто объект есть сейчас (cp-профиль созидателя на ступени 2, прозрачный ящик) → PACK-personal. Стадия = шаг процесса воздействия (что делать в программе на ст.2, форматы, методы) → PACK-<домен>. Аналогия: состояние авто «шасси собрано» ≠ шаг завода «устанавливаем двигатель». Тест: «это про объект или про воздействие на него?»
- **Vault (Волт) ≠ Pack ≠ Файловая система** (Оперативка ИТ, 2026-05-31). Vault = любое хранилище с интерфейсом read/write/find, привязанное к проекту/команде (не к пользователю). Pack = один специализированный тип Vault (онтологический, структурированный по SPF). Embeddings рабочего пространства = Vault IWE, не платформенный сервис. Платформа не тянет данные из репо напрямую — IWE/клиент пушит в платформу. Тест: «поддерживает ли объект read/write/find с доступом по команде — Vault; Pack-структурирован — Pack».

## Конвейер разработки IWE (WP-403)

- **Конвейер IWE ≠ произвольная сессия разработки** (WP-403 Ф1, 2026-06-09). Конвейер = 6 станций (Постановка→Открытие→Проектирование→Работа→Проверка→Закрытие) с фиксированным порядком и обязательным двойным выходом; пропуск станции = незакрытая задача. Произвольная сессия = работа без структуры, выход знания случаен. Тест: «есть ли зафиксированный capture после сессии?»
- **Двойной выход (код + capture) ≠ дополнительный этап после работы** (WP-403 Ф1, 2026-06-09). Двойной выход встроен в станцию Закрытия как инвариант: capture — часть определения «сделано», не опциональный бонус и не отдельная фаза. Тест: «можно ли закрыть задачу без capture?» Да → инвариант нарушен.
- **Пользователь T1–T3 ≠ Разработчик команды T4+** (WP-403 Ф1, 2026-06-09). T1–T3 = пользователи платформы: присылают замечания/фидбек/баг-репорты, но не вливают код и не проходят конвейер разработки. T4+ = команда разработки: сами проходят все 6 станций, делают PR, проходят peer-review. Барьер входа в команду = T4; прогрессия T1→T4 идёт вне конвейера (программы развития, не онбординг WP-403). Тест: «может ли субъект влить PR в код платформы?» Нет → пользователь, не разработчик.
- **Станция конвейера ≠ фаза РП** (WP-403 Ф1, 2026-06-09). Станция = шаг внутри одной задачи (микро-ОРЗ, часы); 6 станций = один прогон. Фаза = крупный блок РП (макро-ОРЗ, дни/недели) с собственным Gate. Тест: «один прогон 6 станций или долгосрочный блок с Gate?» Не путать «Ф1 WP-403» (фаза РП) и «станция 1 Постановка» (шаг конвейера).
- **Dry-run guard ≠ production run.** Dry-run показывает planned actions без side effects — это read-only прогноз. Production run изменяет файлы. Тест: «если убрать флаг --dry-run — изменятся ли файлы?» Нет → dry-run работает. (WP-403-prognoz, peer-session 2026-06-09-18)
- **Постановщик задачи (DP.ROLE.068) ≠ Архитектор задачи (DP.ROLE.069) ≠ Разработчик-исполнитель (DP.ROLE.070).** Три функции в одной задаче: кто решает *что* (Постановщик), *как* (Архитектор), и *делает* (Исполнитель). Могут совпасть в одном человеке, но не сливаться в одну роль. Тест: «может ли решение Архитектора отменить постановку?» Да → разные роли. (WP-403 Ф3, peer-session 2026-06-09-18)
- **Согласование merge (архитектурный gate) ≠ Peer-review (артефактный gate).** Peer-review = проверка по эталону (корректность артефакта). Согласование merge = архитектурное решение + интеграция с roadmap (здоровье системы). Тест: «может ли peer-review быть OK, а merge заблокирован?» Да → разные gate. (WP-403 Ф3, peer-session 2026-06-09-18)
- **Карточка задачи (атом конвейера) ≠ РП (зонт работы).** Карточка = один прогон 6 станций (`inbox/tasks/`). РП = N карточек + фазы. Тест: «один capture на карточку или на весь РП?» На карточку. (WP-403 Ф3, peer-session 2026-06-09-18)
- **Костюм ≠ Оснащение (тир)** (DP.D.131, WP-406 Ф4, 2026-06-09). Оснащение (тир T0-T4) = уровень доступа к среде (какой гардероб открыт). Костюм = конкретная личная сборка под деятельность (персона + инструменты + личное руководство + своя среда). Повышение тира открывает возможность костюма, но не собирает его (это Портной). Тест: «уровень доступа или конкретная личная сборка?»
- **Первокурсник ≠ Участник сообщества** (DP.D.132, WP-406 Ф4, 2026-06-09). Первокурсник = промежуточное состояние входа: закрыты понимание сообщества (Х2) + выбор траектории (Х3), зона Онбордера (DP.ROLE.067). Участник = полная готовность, все 5 характеристик (Х1 оснащён, Х2 понимает, Х3 выбрал, Х4 костюм, Х5 умеет работать). Тест: «закрыты только Х2-Х3 или все пять?»

## Авторские (не затираются update.sh)

> Расширенные версии → `memory/distinctions-warm.md`.

- **Готовая техника ≠ Готовый продукт-рантайм (build-vs-adopt)** (WP-423, peer-session 2026-06-16-21). «Есть готовое» распадается на две вещи. Техника/паттерн (каскад моделей, skill-library, semantic-cache как идея) — переносима, её берут как референс (Ф1 уже взял). Продукт-рантайм (semantic-router, LangGraph, Temporal, GPTCache как сервис) — берут только если он решает ИМЕННО нашу задачу, не навязывает чужую модель управления (coupling против «тонкого клиента») и не упирается в доменный клей, которого на полке нет (наш `executor-catalog.yaml`, хуки, verification-классы). Тест перед «давай возьмём готовое»: «продукт решает нашу задачу или похожую?» (semantic-router = нечёткий NL через эмбеддинги ≠ наш детерминированный гейт по структурным полям); «навяжет ли свою модель состояния, от которой не отвяжемся?»; «знает ли он наш доменный клей?». NIH-самообман ловится проверкой кода: подозрение «зря изобрели велосипед вместо semantic-router» рассыпалось — `iwe_gate.py` это 76 строк stdlib, эмбеддинги не нужны. Каждый «строим сами» снабжать сторож-порогом пересмотра (масштаб 10⁴/мес → движок; вычисляемый confidence → каскады; N повторов → кристаллизация), иначе «keep» застывает в вечный NIH. Связано с [[Доставляемый артефакт ≠ Сервис]], [[суть ≠ интерфейс]] и [[Замена runtime ≠ Замена модели]].
- **LLM-в-центре ≠ ядро-в-центре** (WP-423 Ф3, peer-session 2026-06-16-02). LLM-в-центре = модель держит контекст итерации и дёргает инструменты в петле. Ядро-в-центре = детерминированная программа управляет потоком итераций, LLM вызывается только на планировании или при неопределённости. Тест (D1-b): «управляет ли детерминированная программа потоком итераций рабочего цикла?» Да → ядро-в-центре. Примечание: ядро ≠ конфигурация ядра — конфигурация может создаваться LLM (через кристаллизацию), сам исполнитель детерминирован.
- **Решить задачу ≠ кристаллизовать класс** (WP-423 Ф3, peer-session 2026-06-16-02). Разовое решение = LLM даёт ответ на этот вход, выход эфемерный. Кристаллизация = LLM генерирует исполняемый артефакт (код/правило/хук), сохранённый и запускаемый без LLM на том же классе входов в будущем. Тест: «является ли выход исполняемым артефактом, запускаемым без LLM?» Да → кристаллизация. Ручная и автоматическая кристаллизация равны по тесту — разница во «втором агенте» (человек vs система), не в наличии выхода.
- **Рефлекс ≠ интеллект (runtime-протокол двухконтурного исполнения)** (WP-423 Ф3, peer-session 2026-06-16-02). Рефлекс = детерминированный lookup по сигнатуре признаков в реестре, выполнение программы без LLM. Интеллект = вызов LLM при отсутствии паттерна в реестре; при N повторениях — кристаллизация (D2) + добавление сигнатуры в реестр. Тест D3: «есть ли в детерминированном реестре сигнатура признаков для этого входа?» Да → рефлекс. Нет → интеллект. Паттерн = абстрактная сигнатура признаков (парсинг/структурный тип), не сырой вход. Гейт уверенности = lookup, не LLM-оценка. Три различения взаимосвязаны: D1 (архитектура) + D2 (операция кристаллизации) + D3 (runtime-протокол).
- **Миграционное зеркало (`contract.source=legacy_lms`) ≠ Живой revenue-поток** (WP-390 Ф5, peer-session 2026-06-14-46). Таблица подписок может обновляться сегодня (`contract` db14 Metabase, `max(created_at)`=сегодня) и при этом НЕ быть живым денежным потоком: 706/708 контрактов — перенос легаси-LMS, нативных IWE-контрактов = 1; цены тарифа нет, `contract_event` пуст, `activating_payment_ref` ведёт в замороженный апрельский платёжный дамп (`payment_received` и `finance_payments` оба max ≈ апр-2026, `etl_source=wp268-phase2`). Нового продукта по сути нет: и 526 active-аккаунтов `contract`, и отдельный узкий счётчик `subscribers_snapshot` (=42, db6) — это легаси-миграция, не новые клиенты (две разные метрики разных популяций, не путать). Строить «дашборд выручки» на этом = false-green (выглядит живым, замёрз на апреле). Тест: «таблица обновляется сегодня» ≠ «в ней живые ДЕНЬГИ» — проверь `source` (native vs legacy), наличие цены и потока событий, прежде чем считать это revenue. Уместный живой предмет тут — состояние подписок (adoption tracker), не деньги. Родственно [[HTTP status check ≠ keyword check]] (false-green по форме ответа) и оси legacy→new из «Трек = география ≠ поколение».
- **Программа развития ≠ Primary key выбора руководства** (peer-session 2026-05-31-12, модель пилота, DP.M.241 §S1.5). Программа (ЛР / РР / ИР / …) — runtime-атрибут frontmatter рендера (какой skeleton-шаблон, какой program-overlay подключить) и производное от анализа пилота. Primary key анализа = состояние пилота из 7 источников (digital twin / personal_search / Honcho / LMS+клуб / Диагност / самодекларация / Lifework) + `current_request` priority 1.0. Тест: «программа фильтрует контекст пилота или назначается из контекста?» Назначается → правильно. Фильтрует → анти-паттерн «учебник вместо контекста» (DP.SC.164 §6). При несовпадении `primary_program` ↔ квалификация — `primary_program` побеждает.
- **Governance-фаза ≠ Operations-фаза в зонтике** (peer-session 2026-05-30-40, WP-371). Зонтики не делают governance-работу (контракты, формы, правила) — для governance заводится отдельный РП с владельцем. Operations-работу (метаданные на существующем контенте: разметка, индексация, дополнение графа) зонтик-координатор МОЖЕТ содержать как фазу направления — без отдельного дочернего РП. Тест: «это форма системы (governance) или метаданные на форме (operations)?» Пример: контракт `program × stage × domain` для Портного → WP-371 как отдельный РП. Разметка универсальных руководств по графу понятий → фаза Ф6 WP-364 направления 2 (operations внутри зонтика Фабрики).
- **Суть (ядро IWE-агента) ≠ Интерфейс** (WP-392, директива пилота 6 июня 2026). Суть = ориентация (`IWE_SYSTEM_PROMPT`/`get_instructions`) + знание (`knowledge_search`) + контекст пилота (`get_user_context`/`dt_read`) + память (`personal_search`/ЦД), всё через Aisystant MCP. Интерфейс (бот, **Hermes-рантайм**, Web UI, claude.ai, VS Code) лишь ставит личность пилота, зовёт ядро, рендерит ответ — своей логики знания/памяти не имеет. Инвариант: один вопрос через любой интерфейс → один по существу ответ. Тест: «ответ зависит от того, ЧЕРЕЗ ЧТО спросили (не от формата канала)? → суть протекла в интерфейс». Системный тест: новый экземпляр/канал знает IWE с нулём настройки; если нужно копировать конфиг (`SOUL.md` и т.п.) на каждый рантайм — суть носится копией (анти-паттерн), а не тянется из источника. Обобщение WP-262 («бот = тонкий клиент») на все интерфейсы, включая Hermes.
- **Платформа (сервис-провайдер) ≠ IWE-помощник (клиент)** (WP-415 peer-сессия 2026-06-13, директива пилота). Платформа (Aisystant=мир, MIM=Россия) первична и assistant-agnostic: держит данные/память пользователей (по юрисдикции), универсальные руководства, сервисы генерации; к ней подключается ЛЮБОЙ помощник, не только IWE. IWE — один из клиентов: дистрибутив рабочей среды (Шаблон IWE + паки + методы + локальный шлюз + личный Vault), ставится у человека, ходит в платформу за общими сервисами; не обязательная часть платформы. Ценность IWE = среда + методы + дисциплина структурирования базы знаний (экзоскелет: усиливает мышление, не протез/автопилот); НЕ монополия на данные. Руководства принадлежат платформе, не IWE: универсальный канон → одна организация (русский → российская), персональные генерит платформа из входа (личный IWE). Уточняет [[суть ≠ интерфейс]]: память = сервис платформы через облачный шлюз, не внутри IWE-клиента. Тест: «может ли не-IWE помощник подключиться к платформе и получить те же данные/сервисы?» Да → платформа первична, IWE заменим. SoT-формализация (DP.D + правка WP-392 в манифесте PACK-digital-platform) — после ратификации с Андреем/Ильшатом.
- **Информация (источник) ≠ Канал доставки ≠ Место (поверхность)** (WP-417, директива пилота 11 июня 2026). Одна информация (например, панель показателей пользователя) — единый источник; доставляется разными способами в разные места: на личную страницу через конвейер рендера руководства (как персональное руководство), на открытие дня — чтением, в мессенджер — частью через Доставщик. Канал/место не пересчитывают и не владеют информацией; добавление нового канала не меняет источник. Тест: «если добавить новый способ показать эту же информацию — меняется ли источник?» Нет → правильно (источник один, доставок много). Прямое обобщение [[суть ≠ интерфейс]] с одного интерфейса на множество каналов доставки одной информации. Грань: ось контента (что говорим — «Поле» у Холлидея) ≠ ось доставки (через что/кому — «Канал»): добавить канал ≠ изменить источник, уточнить тон ≠ изменить содержание. Связано с [[Поле ≠ Отношение ≠ Канал]] и [[Голос ≠ Тон]].
- **Источник доставки ≠ Политика-на-место ≠ Физический транспорт** (WP-418, peer-сессия 2026-06-14-39). «Доставка уроков/сообщений/руководства — одна или несколько?» расщепляется на 3 оси. Источников несколько (урок-марафон, нудж, руководство = 3 производителя, не сливать: «что слать» ≠ «как доставить»). Политика доставки одна и привязана к МЕСТУ (Telegram-чат = поверхность, не к источнику и не к каналу-Bot-API): один журнал «что ушло пользователю» (`learning.domain_event`) = источник истины, на нём потолок/дедуп/приоритет/предпочтения — это Доставщик (DP.SC.177). Физических транспортов один-или-несколько = вопрос цены миграции, не правильности: целевое — одна очередь с class-workers, переход — `marathon_queue` остаётся, но пишет общий журнал. Жёсткий потолок видит только capped (must-deliver обходит по определению), поэтому отдельная очередь НЕ обязана консультироваться у Доставщика ради потолка — общее только read-side (журнал). Тест: «потолок „≤N/день“ считает по источнику, по каналу или по месту?» По месту (сколько упало человеку в чат), source-agnostic. Уточняет [[Информация (источник) ≠ Канал доставки ≠ Место (поверхность)]] на стороне транспорта; родственно [[Терминальный fail ≠ Transient fail]] (must-deliver retry) и обещанию честного потолка «≤N необязательных, подписочное сверх».
- **Тело контента (рендер-в-место) ≠ Уведомление о контенте (сообщение)** (WP-418, peer-сессия 2026-06-14-39). Персональное руководство доходит до пользователя как ТЕЛО через рендер-в-место (гит-репозиторий + `/lesson` pull + личная веб-страница, решение WP-301 «руководство → гит-канал»), а в Telegram-чат заходит только УВЕДОМЛЕНИЕ (нудж `nudge_personal_guide_cta`, класс capped) — короткий пинг «занятие готово/руководство обновлено». Гнать тело руководства через очередь сообщений = спутать рендер-в-место с транспортом сообщений. Тест: «человек получает это, открыв сам (pull/страница), или ему прилетает в чат (push)?» Тело — открывает сам; уведомление — прилетает. Долг (найден сессией): мёртвый пуш-путь тела (`<delivery-module>` — ноль живых вызовов после перехода на pull-модель) вычистить или явно переподтвердить гит-канал в Pack, иначе граница держится by-accident, не by-design. Родственно [[Источник доставки ≠ Политика-на-место ≠ Физический транспорт]] и [[суть ≠ интерфейс]].

- **Парламент-модель ≠ Президент-модель** ([systemsworld.club](https://systemsworld.club/t/kak-ustroit-uchyot-personalnyh-dannyh-v-ii-srede-parlament-vmesto-prezidenta/38159), 9 апр; применено WP-73 повестка 18, 17 мая). Профили контроля (Development/Style/Qualification/Onboarding/Domain) — параметр R46, не разные роли. Тест: «решение про приоритет режима?» → R22; «сверка маркера с фактом?» → R46; «диалог с конкретным застрявшим?» → Навигатор; «переключение FSM-состояния в боте?» → R13.

- **Дистрибутив-bundle ≠ Дистрибутив-coupling** (WP-337, 22 мая). Linux-дистрибутив (Ubuntu) = bundle согласованных компонентов + жёсткая связка с ядром (на BSD-ядре Ubuntu не работает). IWE = portable runtime + Pack + host-адаптеры через DP.IWE.011 — bundle есть, coupling с одним LLM-ядром нет. Аналогия: ближе к JVM (portable runtime) + POSIX (контракт хоста), чем к Ubuntu. Тест: «работает ли при смене ядра?» Да → bundle-смысл. Нет → coupling-смысл. Применение в публичных текстах: оговаривать какой смысл имеется в виду.
- **LLM-ядро (платное) ≠ Linux-ядро (open-source)** (WP-337, 22 мая). У Linux-дистрибутива ядро бесплатно — Red Hat-модель «контент бесплатен, инфраструктура платная» работает. У IWE ядро — LLM (Claude, GPT) — внешний платный ресурс, каждый вызов = операционные расходы. Следствие: чистого free-tier нет; T1 реализуется через (а) квоту LLM, (б) BYOK, (в) demo. Применение: при freemium-расчётах учитывать LLM COGS в каждом ARPU; не обещать «бесплатный полный доступ» в trial.
- **Контрольная роль ≠ Операционная роль** (DP.D.080, WP-224 Ф4, 22 мая). Контрольная = наблюдает + отчёт + read-only к аудитуемым артефактам + есть явный получатель (R23-R26). Операционная = изменяет состояние целевой системы + идемпотентность + нет внешнего получателя (R8, R11). Тест 1: «изменились ли аудитуемые артефакты?» Нет → контрольная (output-канал/отчёт не считается). Тест 2: «есть явный получатель отчёта?» Да → контрольная. Смешение: аудитор, правящий то, что проверяет, теряет независимость (2 дефекта); операционная роль, пишущая отчёт вместо действия, = WIP/инвентарь. Разделение логическое, не физическое: один агент может исполнять обе роли в разных сессиях при соблюдении инвариантов.

Диспетчер (DP.ROLE.045) — управляет очередью `inbox/agent/tasks/`, запускает через CCR/systemd. Артефактор-Постановщик (DP.ROLE.058) — нет routing-tag → structured request + тег → возвращает в Маршрутизатор (не вызывает исполнителя напрямую). Декомпозитор (DP.ROLE.053, Ф5 WP-350) — готовый РП ≥3h → этапы с физическими артефактами. Тест: «направить входящий запрос?» → Маршрутизатор; «запустить задачу из inbox?» → Диспетчер; «оформить РП из сырого?» → Артефактор; «разбить РП на этапы?» → Декомпозитор.

- **Агент ≠ Роль** (WP-350/WP-336, 25 мая). Агент = система-исполнитель: {LLM, Инструменты, Промпт, Персона (Память), Контекст} — специализация «Системы» из «Система-в-роли ≠ Роль» (FPF A.2.1, выше). Специализированный: один агент — одна роль, привязка архитектурная; LLM-ядро заменяемо, но роль — нет. Тест: «может ли агент взять другую роль?» Да → универсальный. Нет → специализированный. - **Сторож, метящий просрочку ≠ сторож, самовосстанавливающий её** (neon-backup, 2026-06-09). Watchdog, который при stale-состоянии только инкрементит счётчик провалов, но не запускает обновление, даёт вечный алерт: причина не уходит сама, бэкап застывает на старой дате (4 базы Neon застряли на 27 мая → `failures=4` каждую неделю). - **Доставляемый артефакт ≠ Сервис** (WP-415, peer-session 2026-06-11-22). Артефакт копируется в каждую установку (пак, скилл, база стиля, правило) — к нему применим тир совместности. Сервис = один работающий инстанс, к нему подключаются (бот, MCP-сервер, шлюз, воркер) — тира нет, классифицируется дом × поколение архитектуры × тенантность (single/multi-tenant/per-user). Тест: «объект копируется к пользователю или к нему подключаются?» Чинит аномалию «бот платформенный, но российский»: бот — мультитенантный сервис, не артефакт. SoT: `DS-ecosystem-development/0.OPS/0.9.Inbox/WP-73-object-tiering-model.md`.
- **Трек = география назначения, НЕ поколение продукта** (WP-415, директива пилота 2026-06-11). Трек А = Россия (владелец Ильшат, инфра VK Cloud), Трек В = Мир (владелец Андрей, инфра GKE). Это значения одной оси «дом/назначение». **Оба трека работают на НОВОЙ архитектуре.** Старая (legacy LMS) → новая (IWE) — это ОТДЕЛЬНАЯ ось «поколение», ось миграции: текущая работа Церена = переход старая→новая, цель — оба трека на новой. Подтверждение: WP-281 содержит «Новая архитектура Track A» (Россия тоже переходит на новую). Антипаттерн (склейка в Strategy.md §«Track A vs Track B» от 19 мая): «Track A (Россия, legacy)» связывает Россию с legacy-архитектурой навсегда, «Track B (IWE)» связывает новую архитектуру с одним треком — это смешение двух осей. Тест: «трек — это про страну/инфру или про поколение архитектуры?» Про страну → трек. Про старое/новое → поколение.
- **Тир совместности ≠ Дом/трек ≠ Поколение** (WP-415, peer-session 2026-06-11-22, скорр. 2026-06-11). Три независимые оси. Тир = кто разделяет (платформенный=все пользователи IWE через update.sh / командный=наша команда через командные репо / личный=один участник). Дом/трек = география назначения (Мир=aisystant/GKE, трек В / Россия=mim-school/VK, трек А / канон=iwe-platform / личное). Поколение = старая (legacy LMS) / новая (IWE) — ось миграции, цель «всё → новая». Тир задаётся обещанием (target_tier), не текущим местом файла; разрыв target≠current легитимен только под оформленную задачу промоции со сроком, иначе current=факт.
- **Shadow за отбивающим gate ≠ полное теневое покрытие** (WP-410 Ф-scope, peer-session 2026-06-11-44, ADR-IWE-019). Когда enforcer (шлюз) отбивает deny ДО маршрутизации в бэкенд, теневой in-process guard видит только allow-трафик — запрещённые запросы до него не доходят, паритет deny-путей живым shadow непроверяем. → Двухконтурная верификация: allow-путь — live shadow + офлайн-join по request_id; deny-путь — fixture-сюита (детерминированно, краевые случаи). Тест: «видит ли тень запрещённые запросы, или их режут раньше?» Следствие для rollout: метрика «0 расхождений» покрывает только allow-контур, deny-паритет = зелёные фикстуры.
- **DB-триггер ≠ cross-DB hook** (WP-266 Ф5c, peer-session 2026-06-12-03). Postgres-триггер живёт внутри одной БД и не может звать функции/читать таблицы другой БД Neon (payment/rewards/learning — разные базы). Логика на данных 2+ БД размещается в компоненте с пулами ко всем (projection-worker), не в триггере. Тест: «все таблицы решения в одной БД?» Нет → триггер невозможен, локус = воркер. Прошлая peer-сессия спроектировала триггер, будто базы одна, — проверять размещение таблиц ДО фиксации консенсуса.
- **Поле ≠ Отношение ≠ Канал** (3 оси Холлидея, WP-412 Ф1, peer-session 2026-06-12-02). Куда класть правило языка: доменная лексика (поле) → доменный пак; отношение к читателю (tenor) → пак языковых стилей; канальная адаптация (mode) → интерфейс IWE. Тест: «правило про ЧТО говорим, про КОМУ/КАК или про ЧЕРЕЗ ЧТО?» Операционное «Слот ≠ Overlay» — в README реестра стилей (`PACK-rhetoric/pack/language-style/registry/`), не здесь.
- **Голос ≠ Тон** (WP-412 Ф1/Ф7, peer-session 2026-06-12-02). Голос = стабильная подпись агента, всегда включён, пользователь не крутит. Тон = ситуативная настройка 4 осей (формальность/юмор/уважительность/энтузиазм) — её выбирает пользователь/пресет, и только для человеко-регистров (guardrail Ф7). Тест: «меняется от ситуации → тон; узнаваемо при смене ситуации → голос».
- **Метод входа ≠ Аккаунт поставщика модели** (WP-411 Ф5, 2026-06-11). Метод входа (какой почтой/кодом логинишься в Ory) — ось платформенной идентичности; несколько методов на одну личность — отраслевой стандарт (account linking). Аккаунт поставщика модели (подписка Claude, BYOK-ключи) — ортогонален; привязка метода входа его с личностью НЕ сшивает. Тест: «это про то, КАК человек входит на платформу, или про то, ЧЬЯ LLM отвечает?» Смешение осей → ложный отказ от связки (кейс 2026-06-10).
- **Переподключение клиента ≠ отзыв OAuth-grant** (WP-411 Ф5, тест пилота 2026-06-11). Grant (refresh-token) живёт на стороне клиента (claude.ai) и молча переиспользуется: disconnect/reconnect коннектора окно входа не показывает и личность не меняет. Лечится только отзывом у issuer (Hydra admin API); RFC 7009 revoke требует предъявить сам токен — у пользователя его нет. Тест: «после переподключения показали окно входа?» Нет → grant жив, личность та же.
- **Hydra consent-сессия ≠ Kratos login-сессия** (WP-411 Ф5, peer-session 2026-06-12-01). «Выйти со всех устройств» (Kratos) гасит login-сессии, а refresh-токены/grant'ы Hydra живут независимо — логаут ≠ отзыв OAuth. Kratos settings управляет методами входа, не выданными разрешениями приложений. Тест: «гасим вход человека или пропуск приложения?» Класс ошибок «логаут = отзыв» диагностировать по этой границе.
- **Метод/схема кодирования ≠ Система** (авторское, 2026-06-12). «Система кодирования» — неверный термин: система (FPF) = физический объект, выполняющий функцию в мире. Кодирование и классификация — эпистемическая деятельность (как мы концептуализируем и называем вещи), результат которой — метод или схема, а не система. Правильно: «метод кодирования», «схема классификации», «каталог реестров». Тест: «выполняет ли этот объект функцию сам по себе в физическом мире?» Нет → метод/схема, не система. Применение: WP-419 = «Описание метода кодирования» (верно), не «система кодирования IWE» (неверно).
- **Приёмка обещания (acceptance-фаза) ≠ Прод-наблюдение (passive observation)** (WP-406 Ф8, peer-session 2026-06-12-07). Приёмка проверяет outcome (обещание реально исполняется на живых пользователях): у неё есть метрика + порог N + владелец + триггер закрытия. Наблюдение — пассивный сбор без критерия. Закрыть РП по merged-PR при непроверенном обещании = закрытие по output, не outcome. Приёмка живёт под РП, давшим обещание (OwnerIntegrity: «обещал» и «проверил» — один владелец), не выносится в баг-канал (теряется accountability за метрику) и не в отдельный РП (WIP-инфляция + re-homing обещания). Страховка: если в приёмке всплывает работа уровня разработки — новый РП, не возврат в приёмочную фазу. Тест: «есть метрика+порог+триггер закрытия, и под тем же РП, что дал обещание?» Обобщение [[Inventory ≠ Progress]] на сторону «когда НЕ закрывать». Грань: нудж (push-уведомление) искажает метрику приёмки, если засчитывается как «пользователь получил ценность» — нудж активирует FOMO-механизм, не подтверждает применение метода (аналог «отметил слот» ≠ «понял»). Тест приёмки: «ПРИМЕНИЛ метод или только открыл уведомление?»
- **Снять секрет ≠ снять код, который его требует** (WP-410, peer-session 2026-06-13-02, инцидент 2026-06-13). При cut-over «убрать shared-secret из шлюза ради теста "только адреса"» нельзя снимать секрет/URL раньше, чем удалён fail-closed-код, который их читает. Последствие зависит от ветки: `if (URL && SECRET) {check} else {deny}` → снятие = **fail-closed поломка** (инструмент отдаёт -32000 всем); `if (URL && SECRET) {check}` без else → снятие = **fail-open** (проверка молча пропускается). В WP-410 сняли SCOPE/USER_PROFILE/github раньше кода → 3 семейства инструментов (personal_write, /tier, github_*) сломались в проде, флаг «P1 ПРОЙДЕН» это замаскировал. Правило: перед снятием `X_SHARED_SECRET`/`X_URL` — `grep` обработчиков на `env.X` + обезвредить fail-closed-ветку В ТОМ ЖЕ изменении; authenticated per-tool smoke = gate закрытия каждой cut-over-фазы (не финальный шаг). Тест: «есть ли в коде ветка, которая ломается/обходится, если этого секрета нет?» Да → снимать секрет нельзя. Родственно [[Тесты зелёные ≠ миграция корректна]] (там readers таблиц, здесь readers env) и [[Push ≠ deploy: авто-деплой настраивается per-service]].
- **Mode A бэкенд верифицирует ОБА типа предъявления шлюза (форварднутый user-JWT И gateway-assertion) ≠ только один** (WP-410, peer-session 2026-06-13-10). В Mode A шлюз для jwt-токена форвардит сырой Ory-JWT, для opaque-токена минтит gateway-assertion. Бэкенд, умеющий проверять только assertion (github-integration после Ф7a), молча отвергает весь JWT-путь: `no applicable key found` (Ory-`kid` против JWKS шлюза) → 401 у всех VS Code/CLI/connector-пользователей с 12 июня, незаметно за health-200. Эталон корректного бэкенда — user-profile/learning-context (есть и `verifyOryJwt`, и `verifyGatewayAssertion`). Фикс: либо бэкенд верифицирует оба (P2-направление), либо шлюз всегда минтит assertion для этого бэкенда (Вариант B, gateway `1c05c90`). Тест: «если шлюз пошлёт другой тип предъявления, чем сейчас, — бэкенд его примет?». Родственно [[Снять секрет ≠ снять код, который его требует]] (тот же класс: фактическое предъявление ≠ ожидаемое бэкендом; ловит authenticated per-tool smoke, не health-чек).
- **Co-location генерации с данными ≠ thin client ≠ автономный fallback** (WP-415, peer-session 2026-06-13-28, директива пилота). Один метод генерации руководства исполняется в двух местах (сервис платформы + локально в IWE); место выбирается по правилу «сборка идёт туда, где живут данные; самый приватный участник определяет место сборки». Данные на платформе → генерим на платформе; приватно-локальные (личный Vault) → генерим локально (канон тянется через шлюз, локальное наружу не уходит); микс с приватным → вся сборка локальна. НЕ thin client (IWE не пустой прокси — добавляет локальный контекст) и НЕ fallback (нет второго генератора, метод один). Тест: «есть ли в миксе приватно-локальные данные?» Да → сборка локальна. Скилл переносится как HTTP-сервис платформы + регистрация MCP-инструментом (оба уровня, не «или»). Тот же приём, что [[Local Gateway ≠ Cloud Gateway]]. Обобщение [[суть ≠ интерфейс]] на генерацию: метод = артефакт (копируется), сервис = инстанс метода.
- **Ось юрисдикции (2 организации) ≠ Ось продукта (1 организация)** (WP-415, peer-session 2026-06-13-28). Данные делятся по стране → 2 платформенные организации (`aisystant` мир + российская). Методы/среда не делятся → 1 организация продукта IWE (без юрисдикционного дубля), потому что IWE = доставляемый артефакт-дистрибутив, не сервис с данными. Тест: «объект привязан к данным по юрисдикции?» Да → 2 орг (юрисдикция); нет, копируется к пользователю → 1 орг (продукт). Прямое применение [[Доставляемый артефакт ≠ Сервис]] к топологии организаций.
- **Общее знание (доступ) ≠ Владение source-of-truth (юрисдикция)** (WP-415, директива пилота 2026-06-13). Знание может быть общим **по доступу** (open-source: бери из любой страны, контрибьюторы откуда угодно) и при этом source-of-truth + инфраструктура принадлежат **мировому (не-российскому)** юрлицу. Принцип независимости: мировой контур не зависит от российского ни юридически, ни по инфре, ни по источнику знания. Тест разрываемости (severability): «убрать российскую компанию → мировой продукт работает полностью?» Должно быть «да». Следствие: канон-источник в мире, Россия = downstream-проекция + персональные данные россиян по закону; направление зависимости одностороннее (мир→Россия, никогда Россия→мир для того, что миру нужно). Владение ≠ язык контента (русский канон может принадлежать мировому юрлицу). Уточняет [[Ось юрисдикции (2 организации) ≠ Ось продукта (1 организация)]] и [[суть ≠ интерфейс]]; SoT-формализация в концепции §5a после ратификации с Андреем/Ильшатом.
- **Интеграция как архитектурная конвенция ≠ Интеграция в открытом зонтике** (WP-337 closing, peer-session 2026-06-14-32). Когда зонтик собрал N дочерних РП в единый data-flow (память → наблюдаемость → аналитика), связь между ними НЕ должна держаться фактом «зонтик открыт»: при закрытии её фиксируют кросс-ссылками между живыми РП-владельцами (WP-295 ↔ WP-290) + строкой-конвенцией, иначе через 2 месяца между дочерними появляется шов. Зонтик-контейнер = inventory (мёртвый координационный капитал), конвенция = переносимая истина у владельцев. Тест: «если закрыть зонтик-контейнер, связь между дочерними РП сохранится явной ссылкой или растворится?» Растворится → шов, конвенция не зафиксирована. Родственно [[Inventory ≠ Progress]] и [[OwnerIntegrity]].
- **Локальная сборка зелёная ≠ CI-сборка зелёная (node_modules маскирует stale lock)** (WP-359, peer-session 2026-06-14-31). `npm run <build/package>` использует уже установленные `node_modules` — проходит, даже если `package-lock.json` рассинхронизирован с `package.json` (зависимость добавлена в package.json, lock не пересобран). CI делает `npm ci` (строгая сверка lock↔package.json) → падает на установке ещё до тестов. Локально дефект воспроизводит только `npm ci` (не `npm install`, не `npm run`). Правило: после правки зависимостей — `npm install` + commit lock; верифицировать сборку через `npm ci`, не `npm run`. Тест: «локально зелено, а в облаке красно на установке зависимостей?» → прогнать `npm ci` локально, сверить lock. Родственно [[Тесты зелёные ≠ миграция корректна]] и [[Push ≠ deploy: авто-деплой настраивается per-service]].

## Из установочной встречи практикума «Системное саморазвитие» (14 июня) + пир-сессий

> Доменные (личность/методы воздействия) и инженерные (платформа/правила агентов) — два потока. Краткая форма здесь, развёрнутая формализация (DP.D/PD.D) — в паспортах паков отдельной задачей.

- **Удержание нормы ≠ Поднятие нормы** (установочная встреча СС, 14 июня; корневое для текучка/пожары/развитие). Управление = держать систему в текущей рабочей норме; развитие = перестроить на лучшую норму. Три проекции: работа (текучка ≠ модернизация конвейера), себя (управлять собой ≠ развивать себя), инструмент (ремонт ≠ развитие, спойлер vs замена сломанной детали). Грань: чужие пожары (срочное чужое) вытесняют поднятие нормы → выгорание. Тест: «возвращаю к прежней норме или поднимаю норму выше прежней?» → PACK-<домен>. Связано с [[Состояние объекта ≠ Стадия процесса]] (там объект vs воздействие) и [[Inventory ≠ Progress]].
- **Текущее ≠ Важное ≠ Срочное (ТВС)** (установочная встреча СС, 14 июня, строки 178-358; конвейерная модель). Три режима работы на одном конвейере. **Текущее (текучка)** = штатная работа конвейера по плану/правилам → стабильный плановый результат, приносит деньги; без неё падает производительность. **Срочное (пожар)** = проблема, грозящая ОСТАНОВКОЙ конвейера (или отказ компонента, нужного для работы); цель срочного — вернуть конвейер в плановую норму, поэтому всё бросаем и чиним немедленно. **Важное (развитие)** = модернизация конвейера, чтобы выпускал лучше прежней нормы (больше/быстрее/экономнее); требует осознанно и регулярно выделенного времени («эволюция без выходных»). Тест: «штатный плановый выпуск (текущее) / вернуть остановленный конвейер в норму (срочное) / поднять норму выше прежней (важное)?». Спецслучай: если ВСЁ время уходит на пожары — конвейер морально устарел, его пора перестраивать (важное-развитие), а не латать. → PACK-<домен>. Прямое следствие [[Удержание нормы ≠ Поднятие нормы]] (срочное+текущее = удержание нормы, важное = поднятие); дедлайн ортогонален всем трём ([[Дедлайн ≠ Срочность]]). Производное «срочное своё ≠ срочное чужое»: чужой пожар ложится в твой конвейер извне, без границы «где помогаю, где отказываюсь» вытесняет важное → выгорание (строки 142-145). **Аббревиатура: ТВС.** Канонический источник богаче (универсальное руководство «Методы саморазвития», раздел 9.11 «Как классифицировать работы и составлять бюджет времени» + 3.3 «Важное, текущее и срочное»): три ортогональные оси — Важное/Текущее (развитие vs операционка), Постоянное/Временное (без срока vs с дедлайном) и Плановое/Срочное (запланировано на стратегировании vs внеплановое-сейчас). Срочное = внеплановое восстановление нормы (обычно текучка-класса); дедлайн живёт на оси Постоянное/Временное, поэтому [[Дедлайн ≠ Срочность]] держится структурно. ТВС — учебная свёртка для быстрой разметки; при детальном планировании бюджета времени использовать все 3 оси.
- **Дедлайн ≠ Срочность** (установочная встреча СС, 14 июня, строки 343-357). Дедлайн = согласованный момент передачи работы дальше по конвейеру; **ортогонален триаде** [[Текущее ≠ Важное ≠ Срочное]] — может стоять на текущем, на важном И на срочном (у срочного дедлайн «в течение часа, иначе конвейер встанет»; у важного — «к пятнице сдать раздел»; у текущего — «эту операцию к 15:00, дальше передаю по конвейеру»). Поэтому «есть дедлайн» НЕ равно «срочное» — самая частая ошибка приоритизации. Срочность = ТОЛЬКО угроза остановки конвейера. Тест: «пропущу срок — конвейер встанет (срочное) или следующий этап просто подождёт (дедлайн на текущем/важном)?» → PACK-<домен>.
- **Опыт ≠ Владение методом + SOTA** (установочная встреча СС, 14 июня). Опытный = делал много раз, не осознавая метод; заложник устаревающего ремесла («лучший извозчик, когда все ездят на авто»). Профессионал = осознаёт, что работает методами, знает текущий лучший (state of the art) и переходит на него. Тест: «может назвать свой метод и его SOTA?» → PACK-personal (ступени мастерства). Связано с [[Мастерство (роль) ≠ Степень мастерства (характеристика)]] (FPF A.2, HD #32) и [[Пользователь ИИ ≠ Разработчик собственных агентов]].
- **Личное ≠ Рабочее ≠ Исследовательское развитие** (установочная встреча СС, 14 июня). Личное = взять себя под контроль доступными осознанными методами (SOTA не обязателен). Рабочее = находить и осваивать текущий SOTA в конкуренции. Исследовательское = создавать SOTA на фронтире, «ставить шкуру на кон» (миллиардер сегодня — банкрот завтра). Тест: «организую себя доступным / нахожу лучший существующий / создаю не-существующий метод?» → PACK-<домен> (ЛР/РР/ИР).
- **Достигаторство ≠ Собранность** (установочная встреча СС, 14 июня; ступень 1-2). Достигаторство = глобальные цели и рывки ради объёма (4 часа в пятницу, чтобы «добить раздел»). Собранность = ежедневное удержание внимания на слоте, осознанный вход в роль ученика хотя бы на 5 минут; регулярность важнее объёма. Тест: «гонюсь за объёмом-к-сроку или удерживаю ежедневный слот осознанно? 5 минут каждый день > 7 часов рывком». → PACK-personal. Резонирует с [[Inventory ≠ Progress]] (рывок = ложный прогресс) и [[Учтённое время ≠ Инвестированное время]].
- **Героический режим ≠ Эволюционное развитие** (транскрипты марафона 04, 07, 14 июня 2026; уточнено пилотом 15 июня). Героический режим = ситуативный рывок по необходимости, включается на короткий срок и часто оставляет «похмелье» (мозг запоминает боль и в следующий раз саботирует). Эволюционное развитие = стиль жизни, повседневная практика маленьких шагов, постепенно меняющая норму. Тест: «это ситуативный рывок или стиль жизни, который меняет меня регулярно?» → PACK-personal. Родственно [[Достигаторство ≠ Собранность]] и [[Удержание нормы ≠ Поднятие нормы]].
- **Заметка (видимое) ≠ Нейронные связи (невидимое)** (установочная встреча СС, 14 июня). Заметка = видимый объект-свидетельство; настоящий результат метода (мышление письмом, медленное чтение) = новые нейронные связи, заметка лишь доказывает, что они образовались. Поглощает «прочитать ≠ понять»: цель чтения не объём, а понимание, предъявляемое заметкой. Тест: «что предъявляю (заметка) и что реально изменилось (понимание/связи)? заметка без сдвига понимания = пустой объект». → PACK-personal (методы саморазвития).
- **Замена runtime ≠ Замена модели** (пир-сессия 14 июня). Плоский HTTP-вызов модели (текст→текст) заменяет агентский CLI только для задач без инструментов; нужны файловые операции/commit → рантайм незаменим, HTTP не запускает агентский цикл. Тест: «продукт задачи — чистый текст или файловые операции/commit?» → PACK-digital-platform. Уточняет [[Скрипт ≠ Агент]] и [[Универсальный агент ≠ Специализированный агент]] осью «нужны ли инструменты».
- **Серверный замок (CI/pre-receive) ≠ Локальный хук** (пир-сессия 14 июня). В командном репо локальный commit-хук ловит только того, кто его сам поставил («личный замок в маске командного») — ложное чувство командной защиты. Командный enforcement = серверная проверка при заливке, ловит всех. Тест: «правило сработает у участника, который НЕ ставил себе хук?» Нет → личный замок, не командный enforcement. → PACK-agent-rules.
- **«Почти готово» ≠ «Готово»** (пир-сессия 14 июня). Честный статус различает «чисто by-design» (свойство держится само) и «чисто достижимо после N шагов» (держится на ещё-не-завершённой миграции). Выдержка миграции живого трафика ≠ свойство чистой архитектуры (свежий инстанс рождается чистым). Тест: «свойство держится само или только пока выдерживается миграция?» → PACK-digital-platform. Родственно [[Inventory ≠ Progress]].
- **Декларация (3 строки) ≠ Операционализируемый чеклист (5-7 признаков)** (пир-сессия 14 июня). Чтобы разные агенты размечали пограничные случаи одинаково, ратифицированный текст из трёх строк недостаточен — нужны 5-7 конкретных тестов-признаков. Декларация ≠ тест. Тест: «два разных агента разметят по этому правилу одинаково? проза без признаков → нет». → PACK-agent-rules (формализация AR.* с frontmatter tests).
- **Детектор существует ≠ детектор блокирует** (WP-388 Ф14, 14 июня). `pack-lint.sh` уже умел `exit 1` на ID-коллизии, но обёртка `pre-commit-pack-map` вызывала его ДО `set -euo pipefail` — exit-код проглатывался, детектор работал как warning. Наличие проверки ничего не значит, если вызывающий не уважает её код возврата. Тест: «вызывающий проверяет exit-код проверки (в условии / при включённом set -e), или зовёт «в воздух»?» Зовёт в воздух → warning, не gate. Родственно [[Серверный замок (CI/pre-receive) ≠ Локальный хук]] (там — кто покрыт, здесь — соблюдается ли вердикт).
- **Подсказка показана ≠ ответ затребован (E3-prompted ≠ E3-displayed)** (ADR-IWE-020, peer-сессия 2026-06-14-50, спин-офф WP-419). Non-block nudge (PreToolUse stdout), на который никто не требует ответа, = только *displayed*: агент видит шаблон-подсказку, читает и спокойно продолжает Write, не ответив. Gate реален только если **игнор детектируется** — по заполненным полям в каноне артефакта (PostToolUse presence-check) или в session-warn-log → Close summary. Объявлять «E3-prompted», когда ответа никто не проверяет, = переименование дыры в фичу. Тест: «если агент прочитал подсказку и НЕ ответил — это где-нибудь всплывёт?» Нет → displayed, не prompted. Родственно [[Детектор существует ≠ детектор блокирует]] (там — соблюдается ли exit-код; здесь — требуется ли ответ на показанный nudge) и шкале принуждения E0-E3 (DP.METHOD.054 §4). → PACK-agent-rules / DP.METHOD.054.
- **Замена «выключена» ≠ замена «ошибка» (тихая дыра при снятии секрета)** (WP-410, peer-session 2026-06-15-15). Снять секрет под премисой «функция переехала на замену» опасно вдвойне, если замена не падает, а просто ВЫКЛЮЧЕНА: при «ошибке» smoke краснеет и дыру видно (инцидент 12 июня), при «выключено» smoke зелёный и дыра невидима. В WP-410: S4 снял `SCOPE_SERVICE_SHARED_SECRET` под «принуждение переехало в personal-knowledge», но там `SCOPE_GUARD_MODE=off` (indicators DB недоступна) + шлюз уже снял гейт → ноль активных enforcer'ов, а personal_write зелёный. Тест: «после снятия секрета замена реально ПРИНУЖДАЕТ (видит запрещённое и отбивает) или просто не падает?» Не падает ≠ принуждает. Усиливает [[Снять секрет ≠ снять код, который его требует]]: проверять не «код не сломан», а «функция исполняется», и негативным кейсом (deny-fixture), не allow-smoke. Родственно [[Подсказка показана ≠ ответ затребован (E3-prompted ≠ E3-displayed)]].
- **Явный policy-deny (fail-closed) ≠ infra-сбой стража (fail-open-with-alarm)** (WP-410, peer-session 2026-06-15-16). У authz-стража два класса «не allow», которые НЕЛЬЗЯ сливать в один блок: явный запрет политики (страж достучался до данных и сказал НЕТ) и сбой инфраструктуры (страж НЕ достучался — БД недоступна, таймаут, pool exhausted). Fail-closed на ОБА = единая точка отказа: один флап pooled-эндпоинта блокирует запись ВСЕМ (класс инцидента 12 июня). Правильно: fail-closed только на явный policy-deny, fail-open-with-loud-alarm на infra-сбой. Форма проверки — **positive-list** (блокировать только перечисленные явные deny-reason'ы), НЕ negative-exclude («всё кроме indicators_db_unavailable»): иначе новый тип infra-сбоя (новый reason) молча снова станет SPOF. Реализация WP-410: exhaustive `Record<Reason,'deny'|'infra'>` → новый reason = compile-error, форсит классификацию (личный IWE: `personal-knowledge-service/src/scope.ts shouldBlockOnScope`). Тест: «один кратковременный сбой БД стража заблокирует всех или пропустит с алярмом?» Заблокирует → fail-closed-SPOF. Это fix-сторона к [[Замена «выключена» ≠ замена «ошибка» (тихая дыра при снятии секрета)]]; родственно [[Терминальный fail ≠ Transient fail]] и [[Внутренний алерт (failure) ≠ Внешний heartbeat (dead-man's switch)]].
- **Глобальный детектор ≠ Delta-aware гард** (WP-388 Ф14, DP.SC.181, 14 июня). Глобальный (`check-pack-collisions.sh`: сканирует весь репо, `exit 1` на любой дубль) при уважении exit-кода блокирует ВСЕ коммиты, включая невинные, пока чужой старый дубль не разведён. Delta-aware блокирует только коммит, ВНОСЯЩИЙ новый дубль (`git diff --cached --diff-filter=A`), и подсказывает следующий свободный номер. Тест: «блокирует ли невинный коммит при чужом пред-существующем дубле?» Да → глобальный (collateral); нет → delta-aware. Применение: pre-commit-гард = delta-aware (ловит виновника), CI = глобальный (бэкстоп).
