## Организующий принцип — негативное определение

База **не описывает «хороший код»**. База — это перечень конкретных запахов (антипаттернов) с парой «было → стало». **Вкус — это отсутствие запахов.** Правило, которое нельзя свести к «было/стало», слишком размыто для L0 и уходит в авторский слой L1.

**Зачем база существует (корневая причина).** Агент оптимизирует под «готово сейчас» — у него нет «будущего себя», который будет это поддерживать. Профессионал пишет для следующего читателя (часто себя через 3 месяца). База — встроенный в момент генерации «голос будущего читателя».

## Детектор контекста — «есть ли у кода будущий читатель?»

| Контекст | Признаки | Строгость |
|----------|----------|-----------|
| Боевой / общий код | мёржится в main, живёт в shared-модуле, ожидается тест | строго — все P-правила |
| Одноразовый скрипт / прототип | `/tmp`, разовый анализ, не переживёт задачу | мягко — над-инженерить одноразовое тоже запах |

Тест: «вернётся ли кто-нибудь к этому коду?» Нет → не плоди абстракции. Да → P1-P5 обязательны.

## Граница со смежными слоями

- **Жёсткие запреты** (`git add -A`, секреты в логах, `--no-verify`) → правила агента (PACK-agent-rules, AR.*). Блокирующее, безопасность. **Не дублировать здесь.**
- **Разметка markdown / документы** → `.claude/rules/formatting.md`.
- **База стиля = слой ремесла и вкуса.** Не-блокирующая рекомендация. Тест границы: «ломает систему/безопасность?» → AR. «Делает код хуже на глаз профессионала?» → сюда.

## P0 — триггер-мост (механику закрывает инструмент)

**Перед коммитом кода — запусти форматтер и линтер репо** (`ruff`/`prettier`/`shellcheck`/`eslint` по наличию конфига). Конфига нет → спроси пилота, не выдумывай свой стиль. Это гарантия, что механический слой (отступы, импорты, регистр) закрыт инструментом, а база занимается только craft-слоем.

## P1 — тест без проверки наблюдаемого результата запрещён

«Прогнали и не упало» ≠ «работает». Тест обязан утверждать конкретный наблюдаемый эффект.

**Было:**
```python
await dp.feed_update(bot, update)
assert True
```
**Стало:**
```python
await dp.feed_update(bot, update)
assert len(bot.sent) > 0          # проверяем реакцию, а не отсутствие исключения
```
Запрещено `expect(true).toBe(true)` и `expect(x).toBeDefined()` как единственный assert. Перекликается с «Тесты зелёные ≠ миграция корректна».

## P2 — третье повторение блока → извлечь функцию; не адресовать переменную через `locals()[str]`

**Было:**
```python
for conn_attr in ("rewards_conn", "reference_conn"):
    await locals()[conn_attr].close()   # обращение к переменной по строковому имени
```
**Стало:**
```python
async def ensure_conn(conn, dsn):
    return conn if conn and not conn.is_closed() else await asyncpg.connect(dsn)

conns = {name: await ensure_conn(c, dsn) for name, (c, dsn) in pool_spec.items()}
```
Реальные объекты в списке/словаре, не имена через `locals()`.

## P3 — мёртвая ветка / вариант enum, в который система не переходит, — удалить, не «оставлять для совместимости»

**Было:**
```ts
type CircuitState = "closed" | "open" | "half-open" | "probing"; // half-open никогда не ставится
if (c.state === "half-open") { c.state = "probing"; return true; } // мёртвая ветка
```
**Стало:** убрать `"half-open"` из типа и обработчиков. Мёртвый код «на всякий случай» дороже, чем удалить и вернуть из истории при нужде.

## P4 — `except Exception: pass` без логирования запрещён

Проглоченное исключение прячет первопричину.

**Было:**
```python
try:
    await conn.close()
except Exception:
    pass
```
**Стало:**
```python
try:
    await conn.close()
except Exception:
    log.warning("conn.close failed", exc_info=True)   # видно в логе, не молча
```
Голое `except:` без типа и без действия — всегда запах.

## P5 — длинная функция со смешанными обязанностями → разбить; булевы флаги-режимы → отдельные функции

Сигналы: >~120 строк, >~6 параметров, булевы флаги `skip_x`/`force_y`/`in_z` (control-coupling).

**Было:**
```python
async def process_event(event_id, learning_pool, target_pools, rules,
                        cursor_cache, lookup_pools, event_seq,
                        skip_dlq_check, force_apply, in_replay):
    # чтение + DLQ-проверка + группировка + apply + advance курсора + метрики
```
**Стало:** разнести по обязанностям (`read_event` → `check_dlq` → `group_by_domain` → `apply` → `advance_cursor`); булевы флаги-режимы заменить отдельными вызовами или стратегией. Флаг-режим в сигнатуре = «функция делает две разные вещи».

**Контр-критерий (Ousterhout): не дроби глубокий модуль.** Длина сама по себе — не запах. Глубокий модуль (богатый функционал за узким интерфейсом) лучше мелкого (тонкий интерфейс, сложность вытекает наружу). Дроби, когда внутри **смешаны уровни абстракции** (Мартин) или разные временны́е точки исполнения. НЕ дроби, когда функция длинная, но делает **одно дело на одном уровне абстракции** — разбиение ради длины плодит мелкие хелперы и size-coupling (прыжки по файлу, чтобы собрать одну мысль). Тест: «дробление уменьшает или увеличивает то, что нужно держать в голове, чтобы понять модуль?»

## P6 — обработчик/сервис без структурированного лога на входе и при критичном падении — запах

Чистая структура без наблюдаемости — всё ещё ремесленная ошибка: при инциденте нечем диагностировать. Минимум — структурированная запись на входе обработчика и при критичном отказе.

**Было:**
```python
async def handle(request):
    data = await load(request)
    return process(data)          # упало в проде — почему, откуда, с какими входами?
```
**Стало:**
```python
async def handle(request):
    log.info("[handle] start id=%s", request.id)   # вход виден
    try:
        data = await load(request)
        return process(data)
    except Exception:
        log.exception("[handle] failed id=%s", request.id)  # критичное падение — с контекстом
        raise
```
`[handler]`-prefix локализует источник в общем логе; структурированное поле (id/phase) делает запись машинно-фильтруемой. Не путать с P4: P4 — про проглоченное исключение, P6 — про отсутствие сигнала вообще.
