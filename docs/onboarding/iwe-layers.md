# Архитектура IWE: три слоя

> **Для кого:** для тех, кто уже установил IWE и хочет понять, «как это устроено внутри» и что куда можно менять.

---

## Схема слоёв

```
┌─────────────────────────────────────────────────────┐
│  L3 — Авторское / пользовательское                  │
│  extensions/*.md · params.yaml · memory/ (личное)   │
│  ← НЕ затирается update.sh · это ТВОЁ              │
├─────────────────────────────────────────────────────┤
│  L2 — Staging (обкатка)                             │
│  CLAUDE.md §8 · STAGING.md                          │
│  ← тестируется · при Week Close → L1 или отклонено │
├─────────────────────────────────────────────────────┤
│  L1 — Платформа                                     │
│  .claude/skills/ · memory/protocol-*.md             │
│  CLAUDE.md §1–§7 · .claude/rules/ · .claude/hooks/ │
│  ← обновляется update.sh · трогать осторожно       │
└─────────────────────────────────────────────────────┘
```

---

## L1 — Платформа (не трогай без причины)

**Что здесь:** скиллы (`/day-open`, `/wp-new`, `/archgate`), протоколы (`memory/protocol-*.md`), хуки (`.claude/hooks/`), правила (`.claude/rules/`), §1–§7 в `CLAUDE.md`.

**Кто обновляет:** `update.sh`. Разработчик платформы выпускает новую версию → ты запускаешь `/iwe-update` → L1-файлы обновляются автоматически.

**Аналогия:** ядро ОС. Ты не правишь ядро Ubuntu напрямую — ты устанавливаешь пакеты и настраиваешь конфигурацию.

**Что будет, если отредактировать L1:** при следующем `update.sh` твои правки перезатрутся. Если хочешь изменить поведение — делай это через L3.

---

## L3 — Авторское (твоё пространство)

**Что здесь:** `extensions/*.md`, `params.yaml`, `CLAUDE.md §9` (авторское).

**Кто обновляет:** только ты. `update.sh` не трогает эти файлы — гарантированно.

**Как работают extensions:**

```
extensions/day-open.before.md   ← выполняется ДО шага 0 day-open
extensions/day-open.after.md    ← выполняется ПОСЛЕ шага 6 day-open
extensions/day-close.before.md  ← выполняется ДО day-close
```

**Пример:** хочешь чтобы Day Open проверял задачи в Linear — добавь в `extensions/day-open.before.md`:
```
## Шаг 0a. Проверка Linear
Открыть Linear, посмотреть задачи с дедлайном сегодня.
```

**params.yaml** — глобальные параметры под тебя:
```yaml
author_mode: false        # true = автор шаблона, прямое редактирование L1
governance_repo: DS-strategy   # имя твоего стратегического репо
strategy_day: sunday      # день стратегической сессии
```

---

## L2 — Staging (обкатка перед промоцией в L1)

**Что здесь:** правила на испытании — `CLAUDE.md §8` + `STAGING.md`.

**Когда нужен:** ты придумал новое правило поведения для Claude, протестировал его в работе, хочешь сделать его платформенным (попасть в L1 и в шаблон для всех пользователей).

**Цикл:**
1. Новое поведение → `CLAUDE.md §9` (авторское) + строка в `STAGING.md` (status: testing)
2. Работает стабильно несколько недель → `status: validated`
3. При Week Close → убрать авторские константы → перенести в L1 → коммит в FMT-exocortex-template

## Что куда писать: шпаргалка

| Ситуация | Куда |
|----------|------|
| Хочу изменить поведение Day Open | `extensions/day-open.before.md` (L3) |
| Хочу изменить протокол для всех пользователей | L2 staging → промоция в L1 |
| Моё правило только для меня | `CLAUDE.md §9` (L3) |
| Устойчивое знание обо мне | `memory/*.md` (L3) |
| Нашёл баг в платформенном скилле | `/iwe-bug-report` → issue в FMT-exocortex-template |
| Нужна новая платформенная функция | `/iwe-bug-report` с меткой `enhancement` |

## Почему Syncthing шумит во время сессии Claude

Claude Code непрерывно пишет в `.claude/projects/` (транскрипты сессий) — это сотни файловых событий в минуту. Syncthing реагирует на каждое изменение и сканирует файловую систему.

**Решение:** файл `.stignore` в корне IWE исключает шумные пути:

```
.claude/projects/     // транскрипты сессий Claude Code
.claude/state/
.claude/.cache/
logs/rule-engine/     // журналы rule-classifier (обновляются hourly)
```

Файл `.stignore` уже входит в шаблон — после `/iwe-update` он появится автоматически.
