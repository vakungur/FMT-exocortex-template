# Жёсткие различения (Hard Distinctions)

> **ЗАПРЕЩЕНО путать эти пары.** Критические онтологические ошибки.
> Источник: FPF (A.7 Strict Distinction)
> Архив устаревших различений: `hard-distinctions_archive.md` (WP-7 Ф-2 trim 2026-04-25).

## 1. Персона ≠ Память ≠ Контекст (модель пользовательских данных, DP.D.052)

> Заменяет прежний термин «Цифровой двойник / ЦД» как монолитную сущность.
> Критерий разделения слоёв — **writer + owner (source-of-truth)**.
> SoTA 2025-2026: Letta (persona/human/archival/recall blocks), Mem0 (structured attributes vs raw memories), LangMem (semantic/episodic/procedural), Anthropic Memory tool.

### Три слоя пользовательской модели

| Слой | Writer | Owner (source-of-truth) | Где физически |
|------|--------|-------------------------|---------------|
| **Персона** | пользователь (или агент по его поручению + acceptance) | пользователь — его Git-репо | PACK-personal, DS-strategy, captures, fleeting-notes, preferences (CLAUDE.md, extensions), Ory identity (декларируемые факты) |
| **Память** | платформа автоматически | платформа — Neon | События (activity-hub #3), платежи (#4), расчёты/baseline/potential/indicators (#5 — бывший узкий ЦД), подписки-контракты (#1), проекции паков (#2 индекс) |
| **Контекст** | агент в runtime | runtime (не хранится долго) | Промпт-сборка под один LLM-вызов |

**Тест границы:** «Что пропадёт, если удалю X?» — Git пользователя → Персона. Neon → Память. Прервать LLM-вызов → Контекст.

### Пять категорий **вне** пользовательской модели

| Категория | Содержит |
|-----------|----------|
| **Catalog / Reference** | Тарифы, справочник уровней квалификации (11 ступеней), правила начисления, множители, константы системы |
| **Service / Ops** | Metabase #7, health #8, audit logs, Ory-sessions, Keto-policies, Railway deployments |
| **Relational** | Связи Persona↔Persona: наставничество, community, Random Coffee matching |
| **Proto-Persona** | Pre-Grant, лид до регистрации Ory (email/telegram_id без ory_id); claim flow переводит в Персону |

### Внутри Памяти — расщепление (под-уровни одного слоя)

- **Observed** — события, логи, наблюдения (activity-hub #3, payment events #4)
- Граница: событие ≠ состояние (projection). Одна таблица = один класс (HD Р-W17-1).

### Маппинг старых упоминаний на новую терминологию

| Старое «ЦД» в тексте | → | Новое |
|----------------------|---|-------|
| расчётный профиль (baseline, potential, indicators) | → | **Память.Derived** (~85%) |
| декларация пользователя о себе, роли, ценности | → | **Персона** (~10%) |
| окно LLM / промпт-сборка | → | **Контекст** (~5%) |

### Что осталось от прошлого HD #27 (бот ≠ платформа)

- **Бот = клиент платформы**, не владелец пользовательских данных. Бот пишет события в Память (через `log_event()`), но schema Памяти принадлежит платформе, а операционный session-state бота (`user_state.current_state FSM`) — собственность бота, под-слой Памяти ephemeral, не часть пользовательских данных.
- **Продьюсеры событий** (по значимости): LMS → Клуб → Бот → Web App. Бот — дополнительный источник, основные Observed-события приходят из LMS/Клуба.

**Тест 1:** «Кто владеет этой таблицей?» → платформа (L2) vs бот (L3 instrument). Если платформа → миграция вне бота.
**Тест 2:** «Кто пишет? Кто owner source-of-truth?» → Git пользователя = Персона; Neon платформы = Память; runtime = Контекст.

См. также: DP.D.052 (`.claude/rules/distinctions.md`), `project_persona_memory_context.md`. DP.ARCH.003 расщеплён на DP.ARCH.005/006/007 (WP-257 Ф5 DONE 22 апр).

## 2. Скрипт ≠ Агент (DP.D.048)

| Скрипт | Агент |
|--------|-------|
| **Фиксированный control flow** — автор написал последовательность шагов | **LLM выбирает следующий шаг** на каждой итерации |
| Один вход → одна последовательность действий (результат может отличаться из-за сети/БД) | Один вход → разные последовательности (LLM решает) |
| Вход: аргументы CLI, переменные окружения | Вход: промпт + набор tools |
| Отлаживается как программа (воспроизводимый trace) | Отлаживается как диалог (разные ветвления) |
| Размещение: по типу исполнителя роли (см. ниже) | `.claude/agents/` или `<repo>/agents/` |

**Общее:** оба — исполнители ролей в IWE. Оба могут выдавать разный результат на одинаковом входе (недетерминизм среды: сеть, БД, текущее время). Разница не в предсказуемости результата, а в том, **кто выбирает следующий шаг**: автор скрипта заранее или LLM в runtime.

**Тест:** «Если я уберу LLM из runtime — выполнение сломается?»
- **Да → агент** (без LLM выбирать шаги некому)
- **Нет → скрипт** (упадут только реализации вызовов к LLM, но control flow останется)

**Пример путаницы:** назвать `.claude/hooks/pre-commit-validator.sh` «агентом», потому что он вызывает LLM API. На самом деле последовательность шагов (1. собрать изменения → 2. вызвать LLM → 3. разобрать ответ → 4. решить pass/fail) написана автором скрипта — это скрипт, использующий LLM как tool, а не агент.

**Классификация скриптов по типам (где живёт):**

| # | Тип | Где | Владелец |
|---|-----|-----|----------|
| 1 | Universal protocol (ОРЗ) | FMT/scripts/ → {{WORKSPACE_DIR}}/scripts/ | Платформа |
| 1b | Role-specific protocol | FMT/roles/<role>/scripts/ | Платформа + роль |
| 2 | Author protocol | {{WORKSPACE_DIR}}/scripts/ (не в FMT) | Автор |
| 3 | Service | `<service>/scripts/` | Сервис |
| 4 | Repo operations | `<repo>/.ops/` или `<repo>/scripts/` | Репо |
| 5 | Agent hook | `.claude/hooks/` или `.claude/skills/<skill>/` | Агентная инфра |
| 6 | Git hook | `<repo>/.git/hooks/` или `.husky/` | Репо |

Триггер (launchd/cron/git hook/ручной) — ортогональное измерение, не определяет размещение.

**Правило размещения:** «Чьё обещание ломается, если скрипт упадёт?» = владелец = место. Подробно: DP.KR.001 §5.6.

**Связь:** DP.D.048, DP.KR.001 §5.6, staging S-20 (процесс промоции L3→L1).

## 3. MCP-сервис ≠ Роль ≠ Исполнитель (три слоя)

> Тройное различение, развитое в WP-222 (10 апр → 6 мая 2026). Изначально было «MCP ≠ Роль» (двухслойное), но эмпирический случай Портного показал необходимость третьего слоя — **исполнитель**.

| Слой | Что это | Пример (Портной) |
|------|---------|-----------------|
| **Роль** | Функциональное место в системе. Имеет obligations, методы, контракты. Переносима между носителями. | R27 «Портной» в DP.ROLE.001 — компонует контент под ученика по 8 входам |
| **Носитель / Исполнитель** | Конкретная реализация роли. Может быть Python-скриптом, ИИ-агентом, человеком. Меняется без изменения роли. | Сейчас: `render-pilot-guides.py` + `planner.py` на tsekh-1. Будущее: ИИ-агент tool-use через MCP Gateway |

| ❌ НЕ путать | ✅ Правильно |
|-------------|-------------|
| MCP-сервис = роль (`tailor-mcp` = Портной) | MCP именуется по данным/инструментам; роль — отдельная сущность в Pack |
| Носитель = роль (Python-скрипт = Портной) | Скрипт — носитель R27. Можно заменить на ИИ-агент, роль не меняется |

### Эмпирический пример: жизненный цикл Портного (WP-222 closed 2026-05-06)

| Дата | Событие | Что было ошибкой |
|------|---------|------------------|
| 30 мар 2026 | WP-175 Ф9-A: создан Python stdio-сервис `tailor-mcp` с tool `get_tailor_context` | Имя сервиса = имя роли. MCP-сервис под одну SQL-проекцию. |
| 10 апр 2026 | WP-222 открыт: «зачем нам отдельный tailor-mcp?» — обнаружен слой-разрыв | — (диагностика) |
| 4 мая 2026 | WP-245 Block B: задеплоен реальный носитель — Python-скрипт `render-pilot-guides.py` на tsekh-1; читает Neon `indicators` directly, пишет в `personal-guide` репо. Никто `tailor-mcp` не вызывает. | tailor-mcp оказался zombie-сервисом |

**Тест:** Три вопроса для разделения слоёв:
1. «Это функциональное место в системе с obligations и методами?» → **роль** (в DP.ROLE.001)
2. «Это конкретный код/скрипт/агент, который выполняет роль?» → **носитель** (в DS-autonomous-agents/agents/{role}/)

**Следствия именования:**
- Носителю давать имя с версией реализации: `tailor-py-v0.1` (Python), будущий `tailor-llm-v1.0` (ИИ-агент)
- Роль именуется по функции (Портной, Навигатор) — без привязки к носителю


## 4. Лог ≠ Инцидент ≠ State file (DP.D.049)

| | Log (operational) | Incident (governance) | State file |
|--|-------------------|----------------------|------------|
| **Что** | Хронологический поток событий исполнения | Зафиксированный сбой + разбор причин | Текущее состояние (не история) |
| **Кто пишет** | Автоматически — исполнитель | Человек (или агент под надзором) | Автоматически — скрипт перезаписывает |
| **Формат** | Append-only текст или JSON | Структурированный md: что сломалось, почему, урок | YAML/JSON/текст — снимок состояния |
| **Пример** | `week-draft-append.log`, `payment-worker.log` | P10 (DP.FM.010), P11 (DP.FM.011) | `sync-manifest.yaml`, `last-check.json` |
| **Назначение** | Отладить конкретный запуск | Извлечь урок, попасть в ретроспективу | Следующий запуск (idempotency, дифф) |

**Тест:** «Это хронология, вывод или состояние на момент X?»
- Хронология → **лог**
- Вывод (разбор + решение) → **инцидент**
- Снимок → **state file**

**Антипаттерны:**

1. **Лог в governance-репо.** «Запишу лог выполнения скрипта в `DS-strategy/inbox/`, чтобы потом разобрать» — нарушение OwnerIntegrity. Факт исполнения принадлежит исполнителю, не репо решений. Лог засоряет историю git случайными сообщениями.
2. **Инцидент в `<service>/logs/`.** «Сбой мелкий, запишу прямо рядом с логами» — теряется. Инциденты должны аггрегироваться в governance, чтобы попадать в недельный обзор и стать паттерном (P10, P11).
3. **State file как append-only.** `sync-manifest.yaml` накапливает версии вместо перезаписи — превращается в лог, теряет смысл «текущего состояния». Для истории используется `git log` над самим state-файлом.

**Пример правильной маршрутизации:**

```
{{WORKSPACE_DIR}}/scripts/week-draft-append.sh                     # скрипт
{{WORKSPACE_DIR}}/scripts/logs/week-draft-append.log               # лог (рядом)
{{WORKSPACE_DIR}}/scripts/state/last-append.yaml                   # state file (рядом)
  2026-04-17-week-draft-append-failed.md               # инцидент (если упадёт)
```

**Связь:** DP.D.049, DP.KR.001 §5.7, DP.FM.010 (P10), DP.FM.011 (P11).

## 5. Роль ≠ Агент; Агент ≠ Инструмент; Агентская роль ≠ Функциональная роль (DP.D.023, DP.D.033)

| ❌ | ✅ |
|---|---|
| "Стратег — отдельный агент" | Стратег — **роль** (R1). Claude — holder (исполнитель) |
| "Бот — агент" | Бот — инструмент (I1, Grade 1), играет роль Проводника (R13) |
| "Синхронизатор — агент" | «Синхронизатор» — и роль (R8), и инструмент (I2). Разные ракурсы |
| "Роль требует агентности" | **Агентская** роль (A.13) требует Grade 2+. **Функциональная** (A.2) — нет |

**Role-Centric (DP.D.033):** Роль описывается независимо от исполнителя. Исполнитель (holder) выбирается и подготавливается отдельно. Одно имя = и роль, и система (как «молоток»).
**Нотация FPF:** Holder#Role:Context@Window
**Каталог ролей:** DP.ROLE.001 §3.2 (R1-R20). Реестр исполнителей: §3.1 (A1-A2, I1-I8).
**Подготовка исполнителя:** program (скрипт) | prompt (LLM) | train (человек).
**Тест 1:** Может ли сущность выбирать, КАК действовать? Да → агент. Нет → инструмент.
**Тест 2:** Это описание функции (что делать) или описание системы (чем делать)? Функция → роль. Система → исполнитель.

## 6. Проектировать агента ≠ Проектировать роль агента в контексте (AS.D.007)

| Проектировать агента | Проектировать роль агента в контексте |
|---------------------|---------------------------------------|
| Кто: вендор (Anthropic, OpenAI) | Кто: архитектор/пользователь IWE |
| Что: LLM + runtime + SDK | Что: промпт + контекст + ограничения + governance |
| Артефакт: модель, API | Артефакт: agent-card.yaml, промпт, CLAUDE.md |

**Следствие:** Trust Stack, Graduated Governance, Bounded Agency — применяются к **роли в контексте**, не к агенту. Один Claude Code — разные роли, разный governance.

**Тест:** Ты проектируешь runtime/SDK? → проектируешь агента (задача вендора). Ты проектируешь промпт/границы/governance? → проектируешь роль в контексте (твоя задача).

## 7. Код агента ≠ Данные агента ≠ Решения человека (DP.D.041)

| Код агента (instrument) | Данные агента (governance) | Решения человека (governance) |
|---|---|---|
| Промпты, скрипты, agent-card, trajectory cache | Черновики, отчёты, находки, extraction-reports | Утверждённые планы, подтверждённые captures |
| DS-autonomous-agents | **DS-agent-workspace** | DS-strategy (пример реализации) |
| Меняется редко (версии кода) | Создаётся ежедневно (ночные циклы) | Создаётся при утверждении |
| Автор: разработчик | Автор: агент | Автор: человек |

**Критерий:** Кто автор артефакта? Агент → workspace. Человек → my-strategy. Разработчик → autonomous-agents.

**Trajectory cache ≠ результаты:** trajectory влияет на поведение агента (часть кода, как «обученные веса»), остаётся в instrument-репо. Результаты — потребляются другими, уходят в workspace.

**Pipeline:** Агент создаёт черновик (workspace) → человек ревьюит → утверждённая версия (my-strategy или Pack) → черновик остаётся (аудит, обучение).

## 8. Обязанность → Одна роль → Один сценарий (Anti-Bystander)

| ❌ | ✅ |
|---|---|
| «3 агента чистят WP» | «R1 Стратег, сценарий Session-Prep — единственный владелец архивации WP» |
| Обязанность дублируется в нескольких сценариях | Обязанность принадлежит одной роли, одному сценарию |
| Все должны → никто не делает (bystander effect) | Один владелец → всегда выполняется |

**Правило:** Одна обязанность = одна роль = один сценарий. Другие сценарии могут **помечать** (status: done), но **не выполнять** основное действие.

**Тест:** Если убрать один из N сценариев, выполняющих обязанность — сломается ли что-то? Если нет → дублирование (bystander). Если да → зависимость (ок).

**Пример исправления (21 фев 2026):** Архивация done WP-файлов была в Close (шаг 7), Note-Review (шаг 8), Session-Prep (шаг 8.5). Все три пропускали. Решение: Close и Note-Review только помечают status → Session-Prep перемещает.

## 9. «Уровень» = строго часть-целое (физическое вхождение)

| ❌ НЕ путать | ✅ Правильно |
|-------------|-------------|
| «4 уровня системы» (Ecosystem→Platform→Template→Personal) | «4 контура системы» — организационные контуры, не физическое часть-целое |
| «На каком уровне?» (про контур ответственности) | «В каком контуре?» — контур = граница ответственности и governance |
| «Уровень» = масштаб, зона, контур | **Уровень** = строго U.SystemLevel: уровень 2 физически входит в уровень 1 (как двигатель в автомобиль) |

**Правило:** Слово «уровень» допустимо ТОЛЬКО для физического часть-целого (автомобиль → двигатель → поршень). Для организационных границ, зон ответственности, масштабов → «контур», «зона», «масштаб». НИКОГДА «уровень».

**Тест:** Уровень 2 полностью физически входит в уровень 1? Да → уровень. Нет → контур (или другое слово).

## 10. Platform ≠ IWE Template ≠ Personal IWE (4 контура системы)

| Контур | Что | Governance | Кто меняет |
|--------|-----|-----------|-----------|
| **L2: Platform** | Инфраструктура, сервисы (бот, MCP, DB) | Pack DP, MAP.002 | Разработчик платформы (M2) |
| **L3: IWE Template** | Шаблон для пользователя (CLAUDE.md + memory/ + стратег) | FMT-exocortex-template | Разработчик шаблона (M3) |
| **L4: Personal IWE** | Экземпляр пользователя (настроенный, подключённый) | Личный CLAUDE.md | Пользователь IWE (M4) |

**Тесты:**
- Это затрагивает всех пользователей? → L2 (Platform). Только шаблон? → L3 (Template). Только меня? → L4 (Personal).
- WakaTime: L3 рекомендует (опционально), L4 настраивает.
- Новый MCP-сервер: L2 (Platform) — нужен Pack DP entity + MAP.002.

**IntegrationGate:** Перед интеграцией нового инструмента/агента → определи контур + тип + роли + продукты + процессы. Нет ответов → НЕ устанавливать.

## 11. Репо пользователя ≠ ядро знаний платформы (IWE-архитектура)

| ❌ НЕ путать | ✅ Правильно |
|-------------|-------------|
| «Данные у пользователя» = всё вычислительное | Репо = исходник (Git, Markdown, BYOB). Платформа = вычислительное ядро |
| Пользователь поднимает свой MCP-сервер | Пользователь подключает Aisystant Gateway (один URL, Ory auth) |
| Знания МИМ в личном репо | Знания МИМ (FPF/SPF/ZP/Pack-экосистема) — на платформе |

**Суть:** Репо пользователя — исходник, которым он владеет и может унести. Embeddings, поиск, агенты, ЦД — вычисляются и хранятся на платформе. Пользователь не поднимает локальный MCP-сервер — подключает Gateway как URL. Vendor-agnostic: смена AI-клиента (Claude → ChatGPT) не теряет контекст.
**Тест:** «Если убрать сеть — что у пользователя остаётся?» → только исходники (Markdown). Поиск, агенты, ЦД — недоступны.
**Связь:** WP-158 (C4-диаграммы), WP-189 (MCP-архитектура), ADR-018 (BYOB).

## 12. Pack-контент ≠ Governance-контент

| Pack (source-of-truth) | Governance (управление) |
|---|---|
| **Что истинно** в предметной области | **Что делаем** в проекте |
| Знания, архитектуры, методы | Планы, сроки, реестры |
| Универсально (другой проект) | Локально (этот проект) |

**Тест 1:** Можно ли использовать в другом проекте? Да → Pack. Нет → Governance.
**Тест 2:** "Как устроено?" → Pack. "Что делаем?" → Governance.

## 13. Уровни принципов: 0-й ≠ 1-й ≠ 2-й ≠ 3-й

| Level | Что | Репо-тип | Фреймворк |
|-------|-----|----------|-----------|
| 0 (нулевые) | Транс-дисциплинарные мета-ограничения | Foundation (ZP) | Нет |
| 1 (первые) | Фундаментальные основания дисциплин | Framework (FPF) | = бандл с принципами |
| 2 (вторые) | Принципы предметных областей | Pack | SPF (отдельно) |
| 3 (третьи) | Ситуационные, проектные | Downstream | S2R и другие (несколько) |

**Тест:** Работает в любой области? → 0-й. В любой дисциплине? → 1-й. В конкретном домене? → 2-й. В конкретном проекте? → 3-й.

> Pack = вторые принципы. Downstream = третьи принципы. Подробно: `ZP/README.md`

## 14. Экзокортекс ≠ IWE

| Экзокортекс | IWE |
|---|---|
| Подсистема: память и инструкции (CLAUDE.md + memory/) | Среда: все 5 архитектурных видов (DP.IWE.001 §3) |
| 2 вида: Системы (файлы) + Описания (содержимое) | 5 видов: + Роли + Методы + Рабочие продукты |
| componentOf IWE | Целое |

**Тест:** Затрагивает только файлы инструкций/памяти? → Экзокортекс. Затрагивает роли, методы или рабочие продукты? → IWE.

**Маркетинговый синоним:** «Экзокортекс 2.0» допустим в публичных текстах как синоним IWE. В доменном языке платформы — не использовать.

## 15. Протез ≠ Экзоскелет (паттерн взаимодействия ИИ с человеком)

| Протез (prosthesis) | Экзоскелет (exoskeleton) |
|---|---|
| ИИ заменяет функцию мышления | ИИ усиливает функцию мышления |
| Вход: «Что мне делать?» → Выход: готовый ответ | Вход: «Вот что я думаю» → Выход: дополнение |
| Человек принимает/отклоняет | Человек думает сам, ИИ усиливает |
| Результат: задача решена, человек не научился | Результат: задача решена + человек стал компетентнее |
| Атрофия: навык видеть аффордансы деградирует | Рост: навык видеть аффордансы тренируется |

**Три механизма экзоскелета:**
1. **Предъявление, а не генерация** — ИИ показывает знание пользователя в нужный момент, не генерирует своё
2. **Вопросы, а не ответы** — в стратегических решениях ИИ спрашивает «а ты как думаешь?» до выдачи ответа
3. **Убывающие подмостки (scaffolding)** — помощь уменьшается с ростом компетенции

**Дизайн-принцип IWE (DP.ARCH.001 #21):** IWE — экзоскелет, не протез. Критерий: после взаимодействия пользователь стал компетентнее.

**Тест:** Кто после взаимодействия стал умнее — пользователь или только ИИ получил больше данных?

## 16. Три класса задач по стоимости проверки (Левенчук 2026)

| Класс | Проверка | AI-автономия | Пример |
|-------|----------|-------------|--------|
| **Closed-loop** (тесты) | Дешёвая, автоматическая | Высокая (commoditized) | Код с unit-тестами, математика, криптография |
| **Open-loop** (инженерия) | Дорогая, отложенная | Средняя (люди+агенты) | Марсоход, продукт, реальный SE |
| **Problem framing** | Неизвестная (эволюция) | Низкая (люди ведут) | Стратегия, «развитие для развитых» |

**Следствие для IWE:** Платформа работает на классах 2 и 3. Класс 1 — commoditized, не фокус.
**Следствие для координации:** Класс 2 требует процессного фреймворка (ОРЗ, WP, captures). Класс 3 требует творчества с проверками (семинары, экзокортекс).
**Закон Брандолини:** Чушь предложить легко, опровергнуть — дорого. В классах 2-3 верификация ≥ генерации по стоимости.

**Тест:** Сколько стоит проверить решение по сравнению с его генерацией? Дёшево → класс 1. Дорого → класс 2. Неизвестно → класс 3.

## 17. Верификация ≠ Валидация (VR.D.001)

| Верификация | Валидация |
|---|---|
| «Сделано ли **правильно**?» (соответствие эталону) | «Сделано ли **нужное**?» (пригодность для цели) |
| Эталон: спецификация, Pack-сущность, принципы | Эталон: потребность пользователя, цель |
| Может автоматизировать агент | Чаще требует человека |

**FM:** Подмена верификации валидацией — артефакт «работает» (валиден), но нарушает принципы (не верифицирован).

## 18. Аудит ≠ Ревью (VR.D.002)

| Аудит | Ревью |
|---|---|
| Систематический обход: **полнота** (coverage) | Экспертная оценка: **качество** (depth) |
| Роль: R24 Аудитор | Роль: R25 Рецензент |
| Автоматизируем (Haiku/Sonnet) | Требует экспертизы (Opus/человек) |

**Правило:** Сначала аудит (всё ли есть?), потом ревью (правильно ли?).

## 19. Описание ≠ Знание

| Описание (артефакт) | Знание (capability) |
|---|---|
| В репозитории (FPF, SPF, Pack) | В агенте (человек, ИИ) |
| Копируемо | Требует обучения |
| Проверяется чтением | Проверяется выполнением задачи |

**Тест:** Можно ли передать файлом? Да → описание. Нет → знание.
Подробно: `PACK-personal/.../01B-distinctions.md` D.013, `PD.FORM.004`

## 20. Система ≠ эпистема

| ❌ НЕ путать | ✅ Правильно | Тест |
|-------------|-------------|------|
| Система = область знания | **Система** ≠ эпистема | Можно ли указать на границу в пространстве? |
| "Система мышления" | Системное мышление — это метод, не система | Есть ли у "системы" 4D-границы? |
| "Система управления" = практика | Различай: орг.система vs практика управления | Есть ли владелец, входы/выходы? |

**Частый случай:** «система знаний» — ❌. Знания не являются системой (нет 4D-границ). Корректно: «экзокортекс» (конкретная система) или «набор описаний» (содержимое репозиториев).

**Частый случай:** «система типов» — ❌. Типы — это классификация, не физический объект. Корректно: «типизация», «четыре типа», «классификация репозиториев».

## 24. Проблема ≠ Задача ≠ ФР ≠ Работа (4 уровня, DP.D.053)

> **Главный навык эпохи ИИ:** конвертировать Проблему в Задачу через системное моделирование.
> ИИ-автопилот ускоряет Задача→Работа, но не делает переход Проблема→Задача.

| Уровень | Определение | Характеристика |
|---------|------------|-----------------|
| **Проблема** | Ступор: непонятно, какой метод применить | Требует диагностики и моделирования |
| **Задача** | Понятно какой метод, неизвестен результат | Готово к исполнению |
| **ФР (Формулировка работы)** | Спецификация: исполнитель + метод + срок + РП | Договор между заказчиком и исполнителем |
| **Работа** | Что реально происходит в физ.мире | Может расходиться с ФР |

**Ключевое различие ИИ-эпохи:** 
- ИИ-автопилот (LLM-генератор): решает Задачу → Работа (output возможно ошибочный)
- ИИ-экзоскелет (LLM с human gate): помогает Проблема → Задача (усиливает мышление)

**Тест 1:** Понимаю ли я, какой метод применить? Да → Задача. Нет → Проблема.

**Тест 2:** ИИ ответил на мой исходный вопрос или на упрощённую версию? На упрощённую → Проблема не сконвертирована.

**Подробно:** DP.D.053, .claude/rules/distinctions.md, HD #27

## 25. Контрольная роль ≠ Операционная роль (DP.D.080)

| Контрольная (Control) | Операционная (Operation) |
|---|---|
| Наблюдает, фиксирует, передаёт отчёт | Воздействует, изменяет состояние |
| Read-only к объекту | Write: приводит к норме |
| Инвариант: независимость | Инвариант: идемпотентность |
| Есть явный получатель отчёта | Нет получателя — изменение и есть результат |
| Примеры: R23-R26 (VR), учёт, аудит | Примеры: R8 Синхронизатор, R11 Наладчик |

**Тест:** «Изменилось ли что-то в проверяемой системе после работы роли?»
- Нет → контрольная роль
- Да → операционная

**Почему важно:** Аудитор, исправляющий то, что проверяет, теряет независимость. Операционная роль без изменения — inventory, не throughput. Смешение разрушает оба инварианта.

**Подробно:** DP.D.080, DP.KR.030

## 26. Система-в-роли ≠ Роль (FPF A.2.1, R5:2761)

> Из WP-352 Ф3. Ключевой элемент нотации `система#роль:контекст`.

| ❌ | ✅ |
|---|---|
| «Роль Строителя выполнила работу» | «Иванов в роли Строителя выполнил работу» |
| «Метод определяет, кто работает» | «Роль определяет контекстное участие; работает система в роли» |

**Нотация:** `физическая_система#роль:контекст`. Например: `Иванов#строитель:проект_Альфа`.

**Тест 1:** «Роль что-то сделала?» — Нет. Роль задаёт метод и ожидания. Сама не работает.
**Тест 2:** «Это класс систем или функциональное место?» — Функциональное место → роль.

**Подробно:** FPF A.2.1, `.claude/rules/distinctions.md` (Система-в-роли ≠ Роль)

## 31. Сервис ≠ Метод ≠ Функция (FPF A.6.8, R5)

## 32. Мастерство (роль) ≠ Степень мастерства (FPF A.2, R9:43)
