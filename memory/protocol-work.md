# Протокол Работы (ОРЗ-фрактал)

> **Два масштаба:** День и Сессия.
> **Источник:** CLAUDE.md § 2 (slim) → этот файл.


## § Масштаб: День (Day Work)

> **Действует:** между Day Open и Day Close. Мета-правила для всего дня.
> **Роль:** R1 Стратег (правила) + R6 Кодировщик (исполнение сессий)

### Правила дня

| # | Правило | Суть |
|---|---------|------|
| 1 | **Слот 1 = саморазвитие** | Не переходить к текучке, пока слот саморазвития не отработан или осознанно пропущен. Слот = важное (поднятие нормы); легитимно прерывает его ТОЛЬКО настоящее срочное (угроза остановки конвейера: прод/биллинг упал, блокер для других). Дедлайн, накопившаяся текучка, сильное «горит» — НЕ повод прерывать слот (модель ТВС [[Текущее ≠ Важное ≠ Срочное (ТВС)]] / [[Дедлайн ≠ Срочность]]) |
| 2 | **Сессии = ОРЗ** | Каждая сессия внутри дня — полный цикл Open → Work → Close (§ Сессия ниже). Не пропускать открытие и закрытие |
| 3 | **Помидорки** | Настройки из `memory/day-rhythm-config.yaml` → `pomodoro`. По умолчанию: 25 мин работа → 5 мин перерыв. После 4 помидорок — длинный перерыв 15-30 мин |
| 4 | **Напоминание о перерыве** | Если сессия > `session_alert_minutes` (по умолчанию 50 мин) без перерыва → агент напоминает: *«Пора сделать перерыв. 5 минут.»* |
| 5 | **Сверка с планом** | Между сессиями: «Я ещё на плане дня?» Если нет — осознанное решение, не дрифт |
| 7 | **Resource vs Policy check** | Перед добавлением ресурса (нанять, купить, автоматизировать): constraint выявлен (Identify ✅)? Non-bottlenecks подчинены (Subordinate ✅)? Если нет → вернуться к этим шагам (MIM.FM.010, MIM.FM.011) |
| 6 | **День стратегии** | Проверить `day-rhythm-config.yaml → day_open.strategy_day`. Если сегодня — не пропустить сессию стратегирования |

## 2a. Decision Capture (на каждом рубеже, WP-109 Ф7)

> **Рядом с Capture-to-Pack.** На рубеже, если **пользователь** (не агент) принял решение — зафиксировать.
> **Источник:** PD.SOTA.003 (Decision Register). АрхГейт: 6✅/1⚠️/0❌.

### БЛОКИРУЮЩЕЕ: только решения пользователя

**Пишутся ТОЛЬКО развилки, где выбор сделал пользователь.** Решения агента в рамках своей работы — реализация, структура кода, выбор теста, АрхГейт, который агент проводит сам, — **НЕ писать**. Это работа агента, а не развилка человека.

**Why:** 10 апр 2026 — агент замусорил `decision-log-2026-04.md` 78 записями, из которых только 2 были реальными решениями пользователя. Остальные — операционные/архитектурные решения агента по ходу реализации. Смысл лога — учёт decision fatigue пользователя, а не самоотчёт агента.

**Тест (применять к каждой кандидатной записи):** «Если бы пользователь не сказал этого / не отклонил этого / не выбрал этого — выбор сделал бы сам агент?» Да → **НЕ писать**. Нет → писать.

**Если сомневаешься — НЕ писать.**

### Что пишется

**На рубеже спроси:** *«Было ли решение пользователя?»* Если да — классифицируй и запиши.

| event_type | Когда | cognitive_weight |
|------------|-------|-----------------|
| `decision_approve` | Пользователь подтвердил **нетривиальное** предложение агента, после которого агент без подтверждения не действовал бы | 1 |
| `decision_reject` | Пользователь отклонил / попросил переделать | 3 |
| `decision_redirect` | Пользователь перенаправил задачу (другой РП, другой агент, смена курса) | 3 |
| `decision_architectural` | **Пользователь** выбрал архитектуру / отклонил вариант АрхГейта | 5 |
| `decision_strategic` | **Пользователь** принял решение о приоритетах, scope, бюджете, стратегии | 5 |

**Что НЕ пишется (типовые ошибки агента):**
- Агент провёл АрхГейт сам → не писать (это работа агента)
- Агент выбрал паттерн/реализацию/имя файла/структуру теста → не писать
- Агент обнаружил баг и исправил → не писать
- Агент создал РП по своей инициативе → не писать
- Агент переформулировал фазу, сделал cleanup, реорганизовал код → не писать
- Тривиальные подтверждения пользователя («ок», «да, делай», «норм») — не писать

### Формат и правила записи

**Timezone:** `Asia/Nicosia` (UTC+3). Формат времени: `HH:MM`.

**Timestamp = реальное время** принятия решения, не художественная метка. Сверять с одним из источников: (а) время git-коммита (`git show -s --format=%ai <hash>`), (б) время tool call / сообщения пользователя в текущей сессии, (в) если не можешь сверить — **не писать запись** (правило «если сомневаешься — не писать»). **НЕ писать метки** «ночь», «вечер», «вечер-2/3/5» без привязки к HH:MM.

**Why (10 апр 2026):** в замусоренном логе 78 записей предыдущих сессий были отмечены «ночь 01:30», «вечер-2 21:00», «вечер-5 16:50», но реальные коммиты этих сессий были в 18:49, 12:38 днём и т.д. Агенты писали художественные метки по **порядку добавления записи в файл**, не по фактическому времени события. Это превращает лог в нарратив, а не учётную систему.

**Сортировка:** **последнее сверху**. Новая запись вставляется ВЫШЕ первой существующей секции/записи, не в конец файла.

**Запись (два места, append-only):**
1. **Файл** (сразу на рубеже): `DS-strategy/exocortex/decisions/decision-log-YYYY-MM.md`. Формат строки: `- **HH:MM** [тип/вес] WP-N: Суть решения. Альт: ...`. Read шапку (10 строк) → найти верхнюю секцию `### YYYY-MM-DD` → если сегодняшняя дата есть, вставить новую строку **первой** в секции; если нет — создать новую секцию `### YYYY-MM-DD` **над** всеми существующими секциями.
2. **БД** (при наличии подключения): inline INSERT в `development.user_events` (source=`exocortex`, паттерн Ф3b).

> **БД = авторитетная** для cognitive budget. Файл = для человека. Дрифт допустим.

**Nudge:** после записи — проверить `SUM(cognitive_weight)` за сегодня. Если ≥70% от `cognitive_budget.daily_decision_points` (см. `day-rhythm-config.yaml`) → предупредить:
> *«Сегодня N решений (вес M/порог). Рекомендация: closed-loop сессия или перерыв.»*


## 2b. Verification on milestone (опционально + автотриггер)

> На рубеже, если создан значимый артефакт (Pack-сущность, описание метода, архитектурное решение):

**Вопрос:** *«Артефакт [X] создан. Нужна верификация? (да / на Close / нет)»*

| Ответ | Действие |
|-------|----------|
| **Да** | Запустить `/verify [артефакт]` сейчас |
| **На Close** | Добавить в список для Verification Gate (Close шаг 3b) |
| **Нет** | Пропустить (trivial / closed-loop с тестами) |

> **Не задерживать работу.** Верификация на рубеже — предложение, не блокер. Основная проверка — при Close.

### Автотриггер: после АрхГейта (S56)

> Если в сессии был АрхГейт (ЭМОГССБ-таблица, решение прошло ≥56/70) И после него менялся **код** (не только .md) → Claude **автоматически** запускает `/verify archgate` после завершения реализации.
>
> **Как отличить код от текста:** проверить расширения изменённых файлов. Код = `.py`, `.ts`, `.tsx`, `.js`, `.sh`, `.sql`, `.yaml`, `.yml`, `.json`, `.css`, `.html`. Текст = `.md`. Если только `.md` → верификация кода не нужна.
>
> **Формат анонса:** *«АрхГейт был в этой сессии. Код изменён. Запускаю Верификатора (archgate).»*

> Анонсировать: *«Capture: [что] → [куда]»*

### Творческий конвейер (4 стадии, TTL)

```
Заметка (7д) → Черновик (7д) → Заготовка (14д) → Пост
```

| Стадия | Хранение | TTL | Выходы |
|--------|----------|-----|--------|
| Заметка | `fleeting-notes.md` | 7 дней | captures / черновик / РП / archive |
| Черновик | `DS-strategy/drafts/` | 7 дней | captures / заготовка / РП / archive |
| Заготовка | `DS-Knowledge-Index` status: draft | 14 дней | пост (published) / archive |

> **Черновик ≠ Заготовка.** Черновик — личный (DS-strategy). Заготовка — публичная (DS-Knowledge-Index).
> **Заготовка ≠ Пост.** Заготовка опубликована, но не продвигается. Пост — активно продвигается.
> **Pack → пост:** KE добавил ≥3 сущностей по теме → предложить черновик для поста.
> **Guards:** ≤5 черновиков = норма. 6-10 = ⚠️ предупреждение. >10 = 🛑 блокировка (не добавлять новые).

## 2b. Видео → Captures и Контент (С4+С5)

> **Триггер:** Команда «транскрибируй видео» / `/transcribe <file>` или авто при `video.auto_transcribe.enabled: true` и длительности > `min_duration_minutes`.
> **Зависимость:** скилл `/transcribe` (MLX Whisper, Apple Silicon). Модели: `~/.local/share/mlx-whisper/mlx_models/`.

### С4: Транскрипция → Captures

1. Прочитать `memory/day-rhythm-config.yaml` → `video.auto_transcribe`
2. Транскрибировать через скилл `/transcribe <file>` (по умолчанию `distil-large-v3`, для макс. качества — `large-v3`)
3. Результат → `<video_dir>/transcripts/<filename>.txt`
4. Извлечь captures из транскрипта:
   - Упоминания РП (WP-N) → captures с привязкой к РП
   - Решения, идеи, различения → `captures.md`
   - Анонс: *«Capture (видео): [что] → [куда]»*

### С5: Видео → Посты

> **Триггер:** После С4 (есть транскрипт) + `video.content.suggest_posts: true`.

1. Проанализировать транскрипт на связные рассуждения (>3 мин на одну тему)
2. Предложить зёрна для постов:
   - *«Видео `file.mp4` содержит рассуждение о [тема] (~X мин). Предложить черновик?»*
3. Если пользователь согласен → создать черновик в `DS-strategy/drafts/` по творческому конвейеру (§2)
4. Связать черновик с исходным видео (ссылка в frontmatter)

## 2c. Skill Discovery (на рубежах)

> Рядом с Capture-to-Pack. На каждом рубеже спроси: **есть ли повторяемое действие, которое стоит оформить как скилл?**

**Критерии кандидата в скилл:**
1. Действие повторяется ≥3 раз в разных сессиях
2. Имеет чёткий вход ($ARGUMENTS) и выход (артефакт/решение)
3. Алгоритм устоялся (не меняется от сессии к сессии)

**Если все три → предложить:** *«Skill candidate: [название] — [что делает]. Оформить как `.claude/skills/[name]/SKILL.md`?»*

**Куда:** FMT-exocortex-template/.claude/skills/ (первоисточник) → sync → рабочий {{HOME_DIR}}/IWE/.claude/skills/

## 2d. Review критических документов (Q-bundles + двухуровневый review)

> Для критических Pack-артефактов, архитектурных решений, нормативных текстов.

**Двухуровневый review:**
1. **Executor** создаёт артефакт
2. **Independent reviewer** (другой агент или `/verify` с context isolation) проверяет с правом блокировки

**Q-bundles (явные наборы качеств):**
- Reviewer указывает **конкретные рубрики** проверки, не общие оценки
- ❌ «quality looks stronger» → ✅ «термин X неоднозначен (HD #46), связь с SC-03 не прослеживается»
- Набор рубрик зависит от типа артефакта (Pack-сущность, архитектура, описание метода)

**Когда применять:** Pack-сущности, АрхГейт-решения, описания методов (PROCESSES.md). Не применять к trivial/closed-loop задачам.

**Источник:** ailev, «Процесс для производства AI-агентами текстов» (2026-03-27).
