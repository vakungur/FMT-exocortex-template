# Авторские различения (warm)

> Горячие авторские (4 оперативных) — в `.claude/rules/distinctions.md § Авторские`.
> Этот файл: менее частые, загружается по триггеру.

- **Программа развития ≠ Primary key выбора руководства** (peer-session 2026-05-31-12, модель пилота, DP.M.241 §S1.5). Программа (ЛР / РР / ИР / …) — runtime-атрибут frontmatter рендера (какой skeleton-шаблон, какой program-overlay подключить) и производное от анализа пилота. Primary key анализа = состояние пилота из 7 источников (digital twin / personal_search / Honcho / LMS+клуб / Диагност / самодекларация / Lifework) + `current_request` priority 1.0. Тест: «программа фильтрует контекст пилота или назначается из контекста?» Назначается → правильно. Фильтрует → анти-паттерн «учебник вместо контекста» (DP.SC.164 §6). При несовпадении `primary_program` ↔ квалификация — `primary_program` побеждает.
- **Governance-фаза ≠ Operations-фаза в зонтике** (peer-session 2026-05-30-40, WP-371). Зонтики не делают governance-работу (контракты, формы, правила) — для governance заводится отдельный РП с владельцем. Operations-работу (метаданные на существующем контенте: разметка, индексация, дополнение графа) зонтик-координатор МОЖЕТ содержать как фазу направления — без отдельного дочернего РП. Тест: «это форма системы (governance) или метаданные на форме (operations)?» Пример: контракт `program × stage × domain` для Портного → WP-371 как отдельный РП. Разметка универсальных руководств по графу понятий → фаза Ф6 WP-364 направления 2 (operations внутри зонтика Фабрики).
- **Gateway (паттерн) ≠ Aisystant MCP (инстанс).** Во внешних текстах (consent, UI, README) = «Aisystant MCP».
- **MCP display identity ≠ machine identity.** Display: serverInfo.name, consent UI. Machine: CF Worker name, client_id, env vars. При ребрендинге — только display.
- **Терминальный fail ≠ Transient fail** (scheduler.sh). В retry-loop: терминальный → mark_done (1 TG); transient → retry.
- **Neon pooler маскирует FATAL** как транзиентный (ConnectionDoesNotExistError). Диагноз: повторить через unpooled endpoint.
- **HTTP status check ≠ keyword check** (HD #51). CF Workers: HTTP 200 + 0 bytes = false-green при потере route. Обязателен keyword-check.
- **Single cursor ≠ Per-domain cursor** (WP-270). Single: failure одного projection блокирует все. Per-domain: N строк, изолированы.
- **Smoke-test directory guard ≠ Path validation** (template-sync.sh). Guard detects ephemeral /tmp/iwe-smoke-* pattern + unsets; validation checks file existence after resolution. Оба нужны: guard = prevention, validation = defense-in-depth.
- **Универсальное руководство ≠ Персональное руководство** (WP-300). Оба — проекции Pack PD (SoT). Универсальное: метод для всех, без RCS. Персональное: проекция под пилота через ступень+bottleneck+домен. Не переписывать друг через друга.
- **Local Gateway (DP.IWE.005) ≠ Cloud Gateway (DP.IWE.003 / Aisystant MCP)** (WP-150 Ф6, 11 мая). Local: in-process / Unix socket, single-user, multi-agent в VS Code, обещание = tool routing + file lock. Cloud: HTTPS + OAuth Ory, multi-tenant, для внешних AI-клиентов (claude.ai/ChatGPT), обещание = доставка знаний + access control. Один документ Pack описывать оба = drift через месяц.
- **Peer Agent ≠ Planner/Executor** (WP-150 Ф7, 11 мая). Peer (DP.ROLE.039): равные, координация через explicit-протокол (lock + peer-status + sequential commits), при разногласии → escalation к пилоту. Planner/Executor (отменено): иерархия, один командует, второй исполняет. Тест: если есть «арбитр» из числа агентов → не peer.
- **Нумерация шагов протокола:** буквенные суффиксы (3a/3b/3c) — уникальны в пределах одного файла, не одной секции. Вставка нового шага → дробный формат (3.5), если буква уже занята в другой секции того же файла.
- **Inventory ≠ Progress** (Throughput Accounting, Goldratt). Незавершённый PR, незакрытый РП, draft в inbox = замороженный капитал, не прогресс. Тест: «вышло на пользователя / завершено физически — Throughput; ещё в работе — Inventory». Применение: при оценке «много активных РП» — это сигнал inflated WIP, не продуктивности (WP-313 Ф1).
- **Пользователь ИИ ≠ Разработчик собственных агентов** (WP-245 Ф30, 7 мая). Пользователь: использует готовых агентов через чат/IDE, получает среднее на типовых задачах. Разработчик: пишет DP.ROLE/DP.METHOD в Pack, инженерит MCP-инструменты, собирает Persona/Memory под свою работу. Тест: «формализуешь ли ты собственные методы как контракт для исполнителя?» Развилка — потолок глубины на уровне пользователя; переход на уровень разработчика = качественный сдвиг калибра (ст. 4-5 программы ЛР).
- **Саморазвитие ≠ Работа** (PD.D.142, 4 мая). 4 уровня одновременно: Мастерство (развития себя ≠ работы) / Культура / Методы (M001-M008 ≠ 14 элементов) / РП (ученические ≠ реальные). Тест переносимости: «если завтра поменяешь команду — что останется с тобой, а что в среде?» Остаётся со мной → метод саморазвития. Остаётся в среде → элемент культуры работы.
- **Мировоззрение ≠ Мнение** (PD.FORM.097, 4 мая, WP-245 Ф16). Мировоззрение = исполняемый код (формализованные правила, проверяемые на recall/precision на реальных кейсах), не «позиция». Тест: можно ли проверить рассогласование на конкретном кейсе? Нет → мнение, не мировоззрение. 4 линзы прозрачного ящика — способ читать каноническую модель.
- **Учтённое время ≠ Инвестированное время** (WP-310, 15 мая). Учтённое = всё осознанное время (работа + обучение). Инвестированное = только саморазвитие (slot_logged). Потраченное = учтённое − инвестированное (вычисляется, не спрашивается). stage_evaluator читает только инвестированное.
- **Объект роли ≠ Характеристика роли** (WP-300, 17 мая). Объект = то, на что роль смотрит (проблема для Интеллектуала, задача+продукт для Профессионала). Характеристика = описание качества исполнения (качество мышления — не объект, а мера). Тест: «смотрит ли роль на X или X описывает роль?»
- **Agent Inbox (WP-324) ≠ Session Memory fault-profile (WP-316) ≠ Trace store (WP-295)** (17 мая, уточнено 29 мая). Agent Inbox = транзитная очередь task'ов + результатов. Session Memory = паттерны поведения агента (iwe_memory.db). Trace store = архив решений для replay. Тест: задача/результат vs паттерны поведения vs решения? Роль = функциональное место; исполнитель = любой LLM-агент.
- **Имя «Hermes/Гермес» в IWE НЕ используется для наших артефактов** (29 мая 2026, peer-сессия 2026-05-29-21). Зарезервировано за внешним продуктом **Hermes Agent (Nous Research, <https://hermes-agent.nousresearch.com/>)** — оркестратор LLM-агента с 8 memory providers (Honcho, OpenViking, Mem0, Hindsight, Holographic, RetainDB, ByteRover, Supermemory). Наши роли называются явно: (а) **Session Memory Injector** / «Инжектор сессионной памяти» — выбирает 2-3 паттерна косяков агента из iwe_memory.db и инжектит в начало Day Open; (б) **Сигнальщик инвалидации профиля** (`diagnostician-watcher`, фоновый режим DP.ROLE.042 §3.4) — мониторит триггеры `bh.inv +50%`, `cp_assessment.age ≥90d`, `bh.sys +2` и шлёт `learning.domain_event` Навигатору/Портному/Аттестатору. Зонтик: «**Роли памяти сессий IWE**» (WP-316). Третье значение — **Hermes runtime** в DP.IWE.001 §5.2 — внешний продукт команды Андрея на базе Nous Research Hermes Agent (слой 1).
- **Однозначное result_location ≠ Fallback-цепочка** (WP-324 Ф1, 17 мая). result_location = `repo+branch+path`, единственная точка истины; несоответствие = `status: failed`. Fallback («push в main, иначе PR, иначе ветка»): создаёт навигационный шум (07-10 анализы разбросаны по 4 веткам + 1 PR + main). Тест: «может ли результат оказаться в нескольких местах?» Да → плохой паттерн. **Расширение (WP-324 C-review, 17 мая):** «routing_key + derived path через Карту» = temporal fallback того же класса, если допускает override или Карта изменилась между постановкой и исполнением. Легитимная нормализация требует freeze-at-assignment (диспетчер материализует path в task при `pending→assigned`); без freeze — fail screening ЭМОГССБ по Стабильности. Подробнее → [lessons_temporal_fallback_routing.md](../../memory/lessons_temporal_fallback_routing.md).
- **Memory provider (Honcho-style) ≠ Digital Twin** (WP-316 Ф8, 17 мая). Memory-провайдеры (Honcho, Mem0, Hindsight) производят **нечёткие inferences** из переписки/текста (LLM black-box). Digital Twin (Память.Derived, HD #27) — **детерминированные агрегаты** activity_log по правилам FORM.089 §6.1 (cp.rhy/wld/skl/iwe/int/agt), проверяемые по эталону (VR.R.001). Замена недопустима: разные оси и разный уровень аудита. Тест: «может ли инспектор проверить значение по правилу?» Да → Digital Twin, нет → memory-провайдер. Honcho-style уместен как **дополнительный** слой proxy-метрик из первичного контента (W-рефлексии, captures, переписка) → вклад с весом w_honcho в характеристики со слабым сигналом из activity (cp.wld, cp.agt, bh.awr). Через ArchGate с PII-блокером (B7.3).
- **Парламент-модель ≠ Президент-модель** ([systemsworld.club](https://systemsworld.club/t/kak-ustroit-uchyot-personalnyh-dannyh-v-ii-srede-parlament-vmesto-prezidenta/38159), 9 апр; применено WP-73 повестка 18, 17 мая). Президент: один агент с полным доступом ко всем доменам данных пользователя. Парламент: триада **Учёт / Доступ / Аудит** = N доменных агентов (своя зона ответственности) + 1 координатор (маршрутизирует, не хранит сырьё) + 1 верификатор (изолирован, read-only). Архитектурная гарантия, а не рекомендация — отказ доступа на уровне платформы, не промпта. Тест: «может ли один агент в одиночку прочитать данные не своего домена?» Да → Президент, нет → Парламент. Применяется к разграничению агентов в Agent Workshop Platform.
- **Pack ≠ Программа развития** (WP-334+WP-335, 18 мая). Pack описывает **поле/домен** (онтология, понятия, роли, обещания) — например, PACK-systems-art описывает домен «системное искусство». Программа развития описывает **курс развития конкретной роли** в этом поле — например, Программа Просветительского развития описывает развитие роли Просветителя, опираясь на PACK-systems-art + PACK-personal + PACK-<домен>. Тест: «это описание области знаний или плана развития человека внутри неё?» Не путать при именовании: Pack = существительное-домен (`art`, `personal`, `MIM`), Программа = функция/деятельность (просветительская, личная, рабочая).
- **Контролёр развития ≠ Оркестратор ≠ Навигатор ≠ Проводник** (WP-326/343/346, 20 мая, DP.D.076). **4 близкие роли в области сопровождения пилота**, разные pace-слои и каналы. Контролёр (R46) — pull-sensor, 3× в день, обходит и сравнивает маркер/факт, выдаёт сигнал. Оркестратор (R22) — parent-координатор между N Контролёрами и операционными ролями. Навигатор (MIM.R.007) — мотивационный канал, push-диалог по триггеру (3+ дня тишины + история), не сканирует. Проводник (R13) — реактивный FSM в боте, миллисекунды, исполняет команду «обнови кнопки». Профили контроля (Development/Style/Qualification/Onboarding/Domain) — параметр R46, не разные роли. Тест: «решение про приоритет режима?» → R22; «сверка маркера с фактом?» → R46; «диалог с конкретным застрявшим?» → Навигатор; «переключение FSM-состояния в боте?» → R13.
- **Баллы (gamification) ≠ Бонусы (loyalty) ≠ Ступень (qualification)** (WP-327, DP.D.050, 20 мая). Три системы учёта одной активности. **Баллы** = earned_total (НИКОГДА не убывают) → лидерборд, история; DP.ROLE.034. **Бонусы** = `min(points, Σ(active_days_at_qual_i × daily_cap_i))` с учётом смены квалификации → скидка при оплате, тратятся; DP.ROLE.051 burn. **Ступень** = cp-профиль FORM.089 (13 срезов + 7 bh) → определяет daily_cap. TODO: колонка `earned_total` в `point_balances` (~2h, после WP-327 Этапа 4). Тест: «не убывает?» → баллы. «Тратится на оплату, ограничено историей дней?» → бонусы. «Определяет мастерство?» → ступень.
- **Дистрибутив-bundle ≠ Дистрибутив-coupling** (WP-337, 22 мая). Linux-дистрибутив (Ubuntu) = bundle согласованных компонентов + жёсткая связка с ядром (на BSD-ядре Ubuntu не работает). IWE = portable runtime + Pack + host-адаптеры через DP.IWE.011 — bundle есть, coupling с одним LLM-ядром нет. Аналогия: ближе к JVM (portable runtime) + POSIX (контракт хоста), чем к Ubuntu. Тест: «работает ли при смене ядра?» Да → bundle-смысл. Нет → coupling-смысл. Применение в публичных текстах: оговаривать какой смысл имеется в виду.
- **LLM-ядро (платное) ≠ Linux-ядро (open-source)** (WP-337, 22 мая). У Linux-дистрибутива ядро бесплатно — Red Hat-модель «контент бесплатен, инфраструктура платная» работает. У IWE ядро — LLM (Claude, GPT) — внешний платный ресурс, каждый вызов = операционные расходы. Следствие: чистого free-tier нет; T1 реализуется через (а) квоту LLM, (б) BYOK, (в) demo. Применение: при freemium-расчётах учитывать LLM COGS в каждом ARPU; не обещать «бесплатный полный доступ» в trial.
- **Контрольная роль ≠ Операционная роль** (DP.D.080, WP-224 Ф4, 22 мая). Контрольная = наблюдает + отчёт + read-only к аудитуемым артефактам + есть явный получатель (R23-R26). Операционная = изменяет состояние целевой системы + идемпотентность + нет внешнего получателя (R8, R11). Тест 1: «изменились ли аудитуемые артефакты?» Нет → контрольная (output-канал/отчёт не считается). Тест 2: «есть явный получатель отчёта?» Да → контрольная. Смешение: аудитор, правящий то, что проверяет, теряет независимость (2 дефекта); операционная роль, пишущая отчёт вместо действия, = WIP/инвентарь. Разделение логическое, не физическое: один агент может исполнять обе роли в разных сессиях при соблюдении инвариантов.
- **Маршрутизатор ≠ Диспетчер (DP.ROLE.045) ≠ Артефактор (Постановщик) ≠ Декомпозитор** (WP-350, 25 мая). **4 роли системы исполнения задач IWE**, разные функции. Маршрутизатор (DP.ROLE.059 R30) — единая точка: routing-tag из WP Gate или Артефактора → lookup в executor-catalog.yaml → единственный исполнитель. Диспетчер (DP.ROLE.045) — управляет очередью `inbox/agent/tasks/`, запускает через CCR/systemd. Артефактор-Постановщик (DP.ROLE.058) — нет routing-tag → structured request + тег → возвращает в Маршрутизатор (не вызывает исполнителя напрямую). Декомпозитор (DP.ROLE.053, Ф5 WP-350) — готовый РП ≥3h → этапы с физическими артефактами. Тест: «направить входящий запрос?» → Маршрутизатор; «запустить задачу из inbox?» → Диспетчер; «оформить РП из сырого?» → Артефактор; «разбить РП на этапы?» → Декомпозитор.
- **Workspace-координация peer'ов ≠ Conversational-сессия peer'ов** (WP-337-З-Ф1, 22 мая, DP.D.084). Workspace: разные файлы, parallel, координация через Local Gateway lock + peer-status (DP.SC.034/035). Conversational: одна задача, turn-taking, координация через журнал реплик + маркер `ESCALATE_TO_USER` (DP.SC.154). Внутри conversational — две позиции одной роли DP.ROLE.039: писатель (имеет write-доступ, собирает `report.md`) и напарник (право несогласия и эскалации, не пишет в workspace). Тест: «делят файлы (разные) или обсуждают одну задачу (один отчёт)?» Смешение режимов даёт либо ложные lock-конфликты на одной задаче, либо unilateral-перезапись.
- **Агент ≠ Роль** (WP-350/WP-336, 25 мая). Агент = система-исполнитель: {LLM, Инструменты, Промпт, Персона (Память), Контекст} — специализация «Системы» из «Система-в-роли ≠ Роль» (FPF A.2.1, выше). Роль = функциональное место: что делать, полномочия, домен, I/O. Нотация та же: `агент#роль:контекст`. Тест: «уберу агента — роль остаётся?» Да → роль ≠ агент. Исключение по именованию: специализированный агент (один агент = одна роль, домен ограничен) МОЖЕТ называться именем роли — тогда имя агента = имя роли (session-memory-injector, overnight-reviewer). Универсальный агент (Claude Code, Kimi Code) играет разные роли → имя ≠ имя роли. Детально → [agent-architecture-framework.md](../../memory/agent-architecture-framework.md).
- **Универсальный агент ≠ Специализированный агент** (WP-350/WP-336, 25 мая). Универсальный: может играть много ролей (Claude Code, Kimi Code), широкий контекст, высокий trust. Специализированный: один агент — одна роль, привязка архитектурная; LLM-ядро заменяемо, но роль — нет. Тест: «может ли агент взять другую роль?» Да → универсальный. Нет → специализированный. Примеры специализированных: session-memory-injector, overnight-reviewer, overnight-tester.
- **Платформенный каталог агентов ≠ IWE executor-catalog** (WP-336 vs WP-350 Ф8, 25 мая). Платформенный (WP-336, Андрей): облачные агенты Aisystant, Парламент-модель, HTTPS+OAuth, multi-tenant. IWE executor-catalog (WP-350 Ф8, файл `executor-catalog.yaml`): агенты-исполнители в VS Code, Local Gateway, single-user. Два уровня, два потребителя — не смешивать при проектировании.
- **Session request ≠ Task (Agent Inbox)** (WP-358 peer-сессия, 26 мая). Task (DP.SC.135): отложенный запрос с `due/template/acceptance/result_location`, SLA ≤1ч, batch-обработка. Session request: real-time инициация Claude-сессии извне (Telegram), без `due`, без template, без acceptance — пилот хочет «здесь и сейчас откроем сессию». Тест: «есть ли `due` и `acceptance` или это живой диалог?» Diff в SLA, failure modes, contract. Впихнуть session в `inbox/agent/tasks/` = слом контракта WP-324. Решение: отдельная папка `inbox/agent/sessions/`, один dispatcher с `--mode session`.
- **Session: light ≠ heavy** (DP.SC.162, WP-358 peer-сессия, 27 мая). light = ожидаемый ответ ≤200 токенов, sync SLA P95≤45с. heavy = ответ >200 токенов или неопределённая длительность, async-режим с прогресс-нотификациями каждые 15с. Тест: «ответ уложится в 200 токенов за 45с?» Да → light, sync. Нет → heavy, async. Применение при проектировании SLA для DP.SC.162.
- **Ход (turn) ≠ Сессия** (DP.SC.162 v2, WP-358, 27 мая). Сессия = весь диалог от открытия до `/close` (1..N ходов), идентифицируется `session_id`, живёт в `SESSION-<id>.md` + `SESSION-<id>-thread.md`. Ход = одна пара запрос+ответ, идентифицируется `(session_id, turn_n)`, трекается отдельным domain event. Timeout у сессии (60 мин тишины) и у хода (15 мин hard cut-off) — разные таймеры.
- **Google Calendar «IWE Platform Ops» ≠ process-catalog/ledger ≠ inbox/reminder-*.md ≠ личный Calendar** (DP.D.102, peer-session 2026-05-28-01). Четыре семантических канала событий IWE. One-shot дедлайны РП → Calendar IWE Platform Ops (с мобильным push). Ритмические процессы (day-open, marathon) → process-catalog→ledger→watchdog (автоматика). Дискретные напоминания без автоматики → inbox/reminder-*.md. Личное → личный Calendar. Запись РП-дедлайна в date-ledger.yaml = нарушение OwnerIntegrity (ledger derived, перезаписывается).
- **Создатель DS (Loop A) ≠ Создатель Pack (Loop B)** (SPF-пример, systemsworld.club/t/organizacziya-kastomizaczii..., 2026-05-31). Loop A: доставка ценности конкретному потребителю (адаптация, интеграция под клиента). Loop B: извлечение повторяемых паттернов из локальной работы → масштабируемые компоненты (Pack). Смешение петель = компания выглядит масштабируемой, внутри — ненаправляемый консалтинг. Критическое правило: «local delta не идёт в core roadmap без route decision». Тест: «чья ответственность — доставить этому клиенту или обобщить для всех?»
- **HOT horizon: hard limit 150 lines** (WP-7 NR1.2, 31 мая 2026). Memory-exocortex HOT limit = 150 строк на файл. Превышение = константная ручная перезапись (у Claude Code 1000 линий файла = выше 1000). Скрипт `memory-bleed.sh` проверяет `wc -l`; если >150 → flag. Разбиение: header block (frontmatter + агрегаты) + body block (линейный список); body может быть extracted в отдельный `memory/<topic>-items.md` с `horizon: warm`. Тест: «файл HOT >150 строк?» → split или demote.
- **horizon warm→cold: 14-дневный cutoff** (WP-7 NR1.2, 31 мая 2026). Warm-файлы старше 14 дней (mtime) автоматически понижаются до cold через `sed -i '' 's/horizon: warm/horizon: cold/'` (macOS). Контраст: hot = активная сессия, warm = недавнее (<14d), cold = архив. Скрипт `memory-bleed.sh` выводит кандидатов на demote. Тест: «файл warm, age>14d?» → demote to cold.
- **Внутренний алерт (failure) ≠ Внешний heartbeat (dead-man's switch)** (INCIDENT-2026-06-03). Внутренний (`if: failure()` в самом workflow) ловит «запустилось и упало». Внешний heartbeat (BetterStack ждёт пинг, алертит при тишине) ловит «не запустилось вообще» — отключённое расписание, исчерпанный лимит, выключенный Actions. Аналогия: сторож внутри здания спит, если выключили само здание. Тест: «если процесс не стартует, кто-нибудь узнает?» — только внешний heartbeat. Биллинг-критичный pipeline требует обоих слоёв. Инцидент 12 мая был класса «не запустилось» (sync падал на аутентификации — 3 недели тишины показали: внутреннего алерта мало).
