# Границы ролей и точки передачи (Role Boundaries)

> Артефакт: карта точек передачи между Диагностом (R28), Навигатором (R27), Стратегом (R1) и Плановиком (DP.ROLE.066).
> Scope: кто за что отвечает, когда и как происходит handoff.
> Статус: active

> **Распаковка R1 (РП378, 2026-06-04, peer-консенсус).** Роль R1 ранее сшивала две мыслительные операции: стратегирование (WHAT/WHY) и операционное планирование (HOW MUCH/WHEN). Граница проведена **между ролями**: Стратег (R1) сужается до стратегирования; новая роль Плановик (DP.ROLE.066) принимает операционное планирование. Имя R1 сохранено (Стратег). Discovery-канал стратегирования (запрос пилота из claude.ai) — новая функция, реализуется в РП378 Ф1+.

## Роли

| Роль | Горизонт | Ключевая функция | Триггер |
|------|----------|------------------|---------|
| **Диагност (R28)** | Ступень мастерства, bottleneck | Оценка текущей ступени, выявление bottleneck | Запрос пилота или регулярно (раз в квартал) |
| **Навигатор (R27)** | Квартал, программа | Программное планирование, выбор фокуса квартала | Результат Диагноста + Strategy.md |
| **Стратег (R1)** | Месяц + рефлексия | Стратегирование (WHAT/WHY): discovery НЭП, диагностика состояния, приоритеты месяца | Рефлексия, заметки, сигналы, старт месяца |
| **Плановик (DP.ROLE.066)** | Неделя, день | Операционное планирование (HOW MUCH/WHEN): упаковка приоритетов в РП-неделя, бюджеты, WIP, дни | Контекст приоритетов от R1, старт недели/дня |

## Точки передачи (Handoff Map)

### 1. Диагност → Навигатор

| Поле | Значение |
|------|----------|
| **Триггер** | Завершена диагностика ступени (cp-профиль обновлён) или выявлен bottleneck |
| **Артефакт** | `cp-profile.md` (ступень, сильные стороны, bottleneck, рекомендации) |
| **Формат** | YAML frontmatter + секция «Рекомендации для планирования» |
| **Принимает** | Навигатор — использует bottleneck как вход для выбора фокуса квартала |
| **SLA** | До следующей квартальной сессии стратегирования |

### 2. Навигатор → Стратег

| Поле | Значение |
|------|----------|
| **Триггер** | Утверждён квартальный фокус и программные цели |
| **Артефакт** | `docs/Strategy.md` — секция «Квартал YYYY-QN» (фокус, цели, гипотеза) |
| **Формат** | Markdown с явными R1-R6 квартала, ТОС, гипотеза роста |
| **Принимает** | Стратег — декомпозирует квартал на месяцы, месяцы на недели |
| **SLA** | До первой сессии стратегирования месяца |

### 3. Стратег → Навигатор

| Поле | Значение |
|------|----------|
| **Триггер** | Завершён месяц, итоги по R1-R6 собраны |
| **Артефакт** | `archive/month-reports/Month-Report-YYYY-MM.md` или секция в Strategy.md |
| **Формат** | Факт vs план по R1-R6, отклонения, уроки |
| **Принимает** | Навигатор — корректирует квартальный план при отклонениях >20% |
| **SLA** | В течение 3 дней после Month Close |

### 4. Стратег → Диагност

| Поле | Значение |
|------|----------|
| **Триггер** | Накоплено ≥3 урока, связанных со «ступенью» или «bottleneck» |
| **Артефакт** | `memory/lessons.md` — уроки с тегом `#bottleneck` или `#stage-transition` |
| **Формат** | Markdown, теги для фильтрации |
| **Принимает** | Диагност — использует для следующей диагностики |
| **SLA** | Асинхронно, при следующей диагностике |

### 5. Стратег → Плановик

| Поле | Значение |
|------|----------|
| **Триггер** | Завершена сессия стратегирования месяца (приоритеты + состояние зафиксированы) |
| **Артефакт** | «Контекст приоритетов» — приоритеты месяца + состояние пилота + системные уровни НЭП (не голый `monthly-priorities.md`) |
| **Формат** | Markdown: приоритеты, состояние (Хаос/Потолок/Поворот/Развитие), системные уровни |
| **Принимает** | Плановик (DP.ROLE.066) — упаковывает в РП-неделя с бюджетами + WIP + распределение по дням; имеет право интерпретировать при упаковке |
| **SLA** | До первой сессии планирования недели |
| **Joint ritual** | Стратег участвует в недельной сессии только если приоритеты устарели или состояние изменилось; иначе Плановик ведёт неделю один |

## Что НЕ входит в эту карту

- **Extractor (R25)** — работает с доменным знанием (Pack), не участвует в ритмах стратегирования; handoff в его сторону невозможен — нет интерфейса приёма планов
- **Auditor (VR.R.002)** — вызывается по запросу (per-ArchGate), не является постоянным участником handoff-цикла; инициирует проверку, но не handoff между ролями стратегирования
- **Peer-агенты** — Kimi Code / Claude Code — инструменты, не роли в процессе стратегирования
