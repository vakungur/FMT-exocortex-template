## Главное возражение: `~/.kimi/config.toml` — неверифицированное утверждение

Тезис 3 («Настройка скиллов: `~/.kimi/config.toml` → `extra_skill_dirs = [...]` + `merge_all_available_skills = true`») — это конкретная инструкция с конкретным путём и конкретными ключами. Если хотя бы один элемент неверен, пользователь застревает на шаге 3 без возможности диагностики.

Проблема в том, что я не вижу подтверждения этого формата нигде в FMT-exocortex-template:
- В репо нет директории `.kimi/` — ни в корне, ни в `extensions/`
- `AGENTS.md` описывает Kimi, но не упоминает `config.toml` и `extra_skill_dirs`
- Единственное место, где навыки для Kimi живут — `.claude/skills/kimi-peer-writer/SKILL.md`

Возможны три ситуации: (а) формат верный и это авторская конфигурация, которая не попала в шаблон; (б) формат устарел или относится к конкретной версии Kimi Code; (в) Kimi Code на самом деле читает `.claude/skills/` напрямую как Claude Code, и отдельный `config.toml` не нужен.

**Предложение:** перед написанием KIMI-SETUP.md нужно верифицировать один факт: каким именно механизмом Kimi Code подхватывает кастомные директории скиллов в текущей версии. Если авторский IWE работает — значит, этот config существует у автора. Стоит попросить пилота подтвердить фактический путь и содержимое его `~/.kimi/config.toml`, чтобы не задокументировать догадку как факт.

Если этот шаг неверен — инструкция не работает для 100% пользователей.

## По вопросам писателя

**В1 (объём установки Kimi Code):** Согласен — ссылка + скриншот ключевого шага настройки. Не описывать marketplace-установку самостоятельно.

**В2 (standalone):** Не упоминать standalone вообще. Ссылка на «скопируйте из {{IWE_GOVERNANCE_REPO}}» — это документирование отсутствующего. Это запутает, а не поможет.

**В3 (smoke-тест):** `/kimi-peer-writer --list` — слабый тест. Он проверяет, что Kimi видит скилл, но не то, что скилл работает. Минимальный набор:
1. Kimi вызывает `/kimi-peer-writer --list` → видит сессии (или пустой индекс)
2. `which claude` → Claude CLI доступен (иначе `claude-peer-adapter.sh` упадёт на первом ходу)
3. `cat ~/IWE/.claude/skills/kimi-peer-writer/SKILL.md` → файл существует и читаем

Без проверки Claude CLI peer-сессия сломается не на шаге настройки, а на первом запуске, и пользователь не поймёт почему.

**В4 (SETUP-GUIDE.md):** Да, добавить раздел «Kimi как второй агент» со ссылкой на KIMI-SETUP.md. Это точка входа для большинства читателей.

**В5 (версионирование `config.toml`):** Обязательно. `~/.kimi/config.toml` — персональный файл машины, не репо. Добавить явную пометку и строку в `.gitignore` (если её нет).
